This revision adds the fields that the Solution Architecture Document's decision log required but were missing from the data model: denormalized `branch_ID` on Reservations/Offers/TestDrives (AD-38), nullable `customer_ID` + `guestToken` on Reservations (AD-40), `createdAt`-anchored validity tracking (AD-41), a `PriceHistory` entity (Pricing domain), `Region` on Branches, an `etag`-backing version field on Vehicles (AD-6), and a Data Classification mapping for every new entity (AD-30).

## 1. Purpose

This document defines the CDS data model for the AutoMarket platform: persistent entities, associations and compositions, enums and reusable types, naming conventions, and alignment with the Domain Model — preparing implementation for SAP CAP and SAP HANA Cloud.

## 3. Common Types

File: `db/schema/common.cds`

```cds
using { managed, cuid } from '@sap/cds/common';

aspect BaseEntity : cuid, managed {
}

type CurrencyCode  : String(3);
type Email         : String(255);
type PhoneNumber   : String(50);
```

`managed` provides `createdAt`/`createdBy`/`modifiedAt`/`modifiedBy` on every entity — `modifiedAt` is what backs `@odata.etag` for optimistic locking (AD-6), and `createdAt` is what the Reservation validity window (AD-41) and notification funnels anchor to. Neither needs to be redeclared per entity.

## 7. Pricing Domain *(new)*

File: `pricing.cds`

```cds
entity PriceHistory : BaseEntity {
  vehicle       : Association to automarket.vehicle.Vehicles;
  price         : Decimal(15,2);
  currency      : CurrencyCode;
  effectiveFrom : Timestamp;
}
```

A row is appended whenever `Vehicles.price` changes; the change handler compares against the previous row and emits `VehiclePriceDropped` when the price decreased. `Vehicles.price` itself remains the live, queryable current price — `PriceHistory` is the audit trail behind it, not a replacement for it.

## 9. Reservation Domain

File: `reservation.cds`

```cds
entity Reservations : BaseEntity {
  customer        : Association to Users null;   // null until claimed — see guestToken below
  guestToken      : String(64) null;              // set instead of customer for guest-created reservations
  vehicle         : Association to automarket.vehicle.Vehicles;
  branch          : Association to Branches;      // denormalized from vehicle.branch at creation (AD-38)

  reservationDate : Date;
  notes           : String(1000);

  status          : ReservationStatus;
}

type ReservationStatus : String enum {
  REQUESTED;
  APPROVED;
  COMPLETED;
  CANCELLED;
  EXPIRED;
};
```

**Constraints**:

* Partial unique index: `UNIQUE (vehicle) WHERE status IN ('REQUESTED', 'APPROVED')` — backs Domain Model Rule R-4 at the DB level, not just in application code.
* Either `customer` or `guestToken` must be set, never both null (enforced in the repository, not expressible as a plain CDS constraint).
* `createdAt` (from `BaseEntity`/`managed`) is the anchor for the 48-hour validity window (Rule R-7) — the expiry poller reads `createdAt + 48h`, never a value that resets on a failed checkout attempt.

`branch` denormalization exists so the ABAC `where` clause in the Authorization Matrix evaluates against a direct column instead of a join through `vehicle` on every read.

## 10. Test Drive Domain

File: `test-drive.cds`

```cds
entity TestDrives : BaseEntity {
  customer      : Association to Users;
  vehicle       : Association to automarket.vehicle.Vehicles;
  branch        : Association to Branches;   // denormalized, same rationale as Reservations
  scheduledDate : Date;
  scheduledTime : Time;
  status        : TestDriveStatus;
}

type TestDriveStatus : String enum {
  REQUESTED;
  APPROVED;
  COMPLETED;
  CANCELLED;
};
```

Unlike Reservations, `customer` here is **not nullable** — a Test Drive never carries an ownership guard later in its lifecycle, so it needs no guest-token/claim mechanism even though Guest Checkout permits the initial request. Guest contact details for an unauthenticated test-drive request are captured inline at booking time, outside this entity (see OData Service Design Document).

## 13. Payment Domain

File: `payment.cds`

```cds
entity Payments : BaseEntity {
  order                 : Association to Orders;

  provider              : String(50);

  amount                : Decimal(15,2);
  currency              : CurrencyCode;

  transactionReference  : String(255);
  idempotencyKey        : String(255);

  status                : PaymentStatus;
}

type PaymentStatus : String enum {
  INITIATED;
  AUTHORIZED;
  CAPTURED;
  FAILED;
  REFUNDED;
};
```

Unique constraint on `idempotencyKey`. **No PAN/CVV field exists on this entity, by design** (AD-4) — `transactionReference` is the PSP's own reference, never a card number; AutoMarket's PCI-DSS scope stays at SAQ-A because no card data is ever modeled here.

## 17. Event Outbox

File: `audit.cds`

```cds
entity EventOutbox : BaseEntity {
  eventType      : String(255);
  aggregateId    : UUID;
  branchId       : UUID;     // carried in the envelope for branch-scoped consumers
  payload        : LargeString;
  published      : Boolean;
  publishedAt    : Timestamp;
}
```

Written in the **same transaction** as the business state change (transactional outbox, AD-5). A poller dispatches unprocessed rows at-least-once; consumers (Notification, Audit, Reporting, Search index updater) must be idempotent. Retention: 2 years for the raw rows — Reporting's own derived projections (Section 18) are **not** subject to this window.

## 18. Reporting Domain

Reporting entities are read models, populated asynchronously from `EventOutbox` — never queried live against the operational tables above. No business logic may depend on reporting entities.

```text
VehicleSalesReport
OfferConversionReport
ReservationReport
BranchPerformanceReport
```

These projections persist independently of `EventOutbox`'s 2-year retention (see Section 17).

## 20. Technical Rules

| Rule | Statement |
|---|---|
| CDS-1 | All entities must inherit from `BaseEntity`. |
| CDS-2 | All enums must be declared in domain-specific files. |
| CDS-3 | Business logic must not be implemented inside CDS entities. |
| CDS-4 | All domain validations must be implemented in the Domain Layer. |
| CDS-5 | CAP Services must consume CDS entities through service projections — never raw entities. |
| CDS-6 | Every business invariant stated as "only one active X per Y" in the Domain Model must have a corresponding partial unique index here — a rule with no DB-level backing is a race condition waiting to happen (see Sections 9 and 12). |

## 21. Data Classification

A mapping from the Solution Architecture Document's classification tiers to the entities above — added because new fields were being introduced (Section 7, 9) with no single scheme to classify them against:

| Tier | Entities / fields |
|---|---|
| Public | `Vehicles` (catalog-facing fields), `VehicleImages` |
| Internal | `Branches`, Reporting projections |
| Confidential | `Users` (PII fields), `Notifications.content` |
| Restricted | `Payments` (`transactionReference`, `idempotencyKey`), `AuditLogs` (PII-bearing rows), `EventOutbox` rows carrying a `customer`/`guestToken` reference |

`EventOutbox` and `Reservations.guestToken` are explicitly called out here because they were the two entities most likely to be overlooked in a PII audit — neither looks like a "customer data" table at a glance.
