## 1. Purpose

This document defines authorization rules for the AutoMarket platform: role permissions, action permissions, data access restrictions, branch-based access rules, ownership rules, and the foundation for XSUAA roles and CAP authorization.

---

## 2. Authorization Strategy

### Role Based Access Control (RBAC)

Roles: Admin, Manager, Operator, Customer.

### Guest Access (not a role — a pre-authentication state)

Per Guest Checkout, an unauthenticated visitor may browse, create a Reservation, and request a Test Drive. A guest is identified by a signed `guestToken` (Reservation) or by contact details captured inline (Test Drive) — never by an XSUAA/IAS session. A guest's Reservation must be **claimed** (authenticated or registered) before checkout can proceed; Offer submission is never available to a guest.

### Attribute Based Access Control (ABAC)

Additional restrictions via attributes: branch assignment, resource ownership, vehicle status, reservation ownership.

---

## 3. Role Definitions

### Admin

System administrator. Scope: **All Branches, All Data**. MFA: **mandatory**.

### Manager

Business manager: offer approval, reservation supervision, sales monitoring, branch performance management. Scope: **Assigned Branches**. MFA: **mandatory**.

### Operator

Operational employee: vehicle management, reservation handling, test drive management. Scope: **Assigned Branch Only**. MFA: **mandatory**.

### Customer

External customer: browse, favorites, reservations, offers, purchase. Scope: **Own Data Only**. MFA: **risk-based / optional at launch** (mandatory MFA here would conflict with the Guest Checkout friction-reduction goal).

### Guest

Unauthenticated visitor. Scope: **own guest-token-bound resources only**, and only for the actions listed in Section 4/6/7. No MFA concept applies — a guest has no account.

## 6. Reservation Module Permissions

| Action | Admin | Manager | Operator | Customer | Guest |
|---|---|---|---|---|---|
| View Reservations | ✅ | ✅ Branch | ✅ Branch | Own | Own (via `guestToken`) |
| Create Reservation | ❌ | ❌ | ❌ | ✅ | ✅ |
| Cancel Reservation | ✅ | ✅ Branch | ✅ Branch | Own | Own (via `guestToken`) |
| Claim Reservation *(new)* | — | — | — | ✅ (at checkout) | n/a |
| Approve Reservation | ✅ | ✅ Branch | ✅ Branch | ❌ | ❌ |
| Reject Reservation | ✅ | ✅ Branch | ✅ Branch | ❌ | ❌ |
| Complete Reservation | ✅ | ✅ Branch | ✅ Branch | ❌ | ❌ |

**ABAC Rule**: Operators/Managers may only manage reservations whose denormalized `branch` column matches their assigned branch — evaluated directly, not via a join through Vehicle (AD-38).

**Guest access rule**: a guest may read or cancel a reservation only by presenting the `guestToken` returned at creation — this is a possession check, not an identity check, and is independent of XSUAA/IAS. At `CheckoutStarted`, the guest must authenticate or register (Claim Reservation); from that point all rows above apply exactly as written for Customer.

## 17. Authentication Hardening

| Control | Policy |
|---|---|
| MFA — Operator / Manager / Admin | **Mandatory** |
| MFA — Customer | Risk-based / optional at launch |
| Account lockout | 5 failed attempts within 15 minutes → 30-minute time-based lockout |
| Session idle timeout | Admin 15 min, Manager/Operator 30 min, Customer follows IAS default (enforced at the Approuter) |
| Password hashing | Bcrypt |

## 18. Rate Limiting

Closing a gap this matrix previously had no row for: Guest Checkout permits unauthenticated writes (Section 6, 7), but no limit had ever been attached to that traffic.

| Scope | Limit | On breach |
|---|---|---|
| Authenticated user — writes | 100 req/min | `429` + `Retry-After` |
| Authenticated user — reads | 300 req/min | `429` + `Retry-After` |
| Anonymous/guest — writes (Reservation, Test Drive) | 20 req/min per IP | `429` + `Retry-After` |
| Anonymous/guest — reads | 100 req/min per IP | `429` + `Retry-After` |

Guest limits are intentionally stricter and IP-based, since anonymous traffic has no account to hold accountable.

## 19. CAP Authorization Mapping

```text
AutoMarket.Admin
AutoMarket.Manager
AutoMarket.Operator
AutoMarket.Customer
```

A **guest** never maps to an XSUAA role — the `guestToken` check happens in a custom handler before any `@restrict` evaluation runs, since there is no token to evaluate against.

Example — Vehicle Service:

```cds
@restrict: [
  { grant: ['READ'], to: 'AutoMarket.Customer' },
  { grant: ['*'],    to: 'AutoMarket.Admin' }
]
```

Example — Reservation Service, branch-scoped (AD-3, AD-38):

```cds
@restrict: [
  { grant: ['READ'], to: 'AutoMarket.Customer', where: 'customer_ID = $user' },
  { grant: ['*'],     to: 'AutoMarket.Operator', where: 'branch_ID = $user.branch' },
  { grant: ['*'],     to: 'AutoMarket.Manager',  where: 'branch_ID = $user.branch' },
  { grant: ['*'],     to: 'AutoMarket.Admin' }
]
```

`branch_ID` is the denormalized column from the CDS Data Model — this `where` clause is the entire reason that column exists; without it, every read would need a join through Vehicle.

Offer Service action example:

```cds
action approveOffer()
```

Allowed Roles: `Admin`, `Manager`.

## 20. ABAC Rules

| Rule | Statement |
|---|---|
| ABAC-1 | Operators may only access vehicles belonging to their branch. |
| ABAC-2 | Managers may only access branch-specific operational data. |
| ABAC-3 | Customers may only access their own Reservations, Offers, Orders, Payments, Test Drives, Favorites. |
| ABAC-4 | Customers may only modify resources they created. |
| ABAC-5 *(new)* | A guest may only access a Reservation by presenting its `guestToken`; possession of the token is the entire authorization check until the reservation is claimed. |
