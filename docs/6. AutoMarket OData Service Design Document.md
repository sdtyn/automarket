## 1. Purpose

This document defines all OData services exposed by the AutoMarket platform: service boundaries, exposed entities, actions, functions, API ownership, authorization requirements, and frontend consumption contracts.

---

## 2. Service Architecture Principles

* One domain = one service.
* Services expose projections only — no direct database entity exposure.
* Actions contain business operations; functions contain read-only operations.
* Authorization is enforced at service level (Authorization Matrix).
* Rate limiting is enforced one layer up, at the Approuter (Section 18) — services themselves assume traffic already passed that gate.

File: `modules/identity/api/identity-service.cds`

**Exposed Entities**: `Users` (read only), `Roles` (read only).

* `updateProfile` — `{ firstName, lastName, phoneNumber }`

**Authorization**:

| Action | Roles |
|---|---|
| Read Own Profile | All |
| Update Own Profile | All |
| Manage Users | Admin |

## 5. Branch Service *(new)*

File: `modules/branch/api/branch-service.cds`

**Exposed Entities**: `Branches`.

**Functions**: `getActiveBranches` — returns active branches only, used by Vehicle creation and the catalog's branch filter.

**Authorization**: View — Admin/Manager/Operator; Create/Update/Disable — Admin only (see Authorization Matrix Section 16).

---

## 6. Vehicle Service

File: `modules/vehicle/api/vehicle-service.cds`

**Exposed Entities**: `Vehicles`, `VehicleImages`.

**Actions**:

* `createVehicle` — Allowed Roles: Admin, Manager, Operator.
* `publishVehicle` — `DRAFT → FOR_SALE`.
* `archiveVehicle` — `FOR_SALE → ARCHIVED`.
* `updateVehiclePrice` — Allowed Roles: Admin, Manager. Delegates to PricingService internally to append a `PriceHistory` row and evaluate `VehiclePriceDropped`.

**Functions**:

* `getAvailableVehicles` — returns `FOR_SALE` vehicles only.
* `getVehicleDetails` — returns complete vehicle details.

**Concurrency** (AD-6): `Vehicles` is exposed with `@odata.etag: modifiedAt`. Every status-changing action requires the caller's `If-Match` header to match, or returns `412 Precondition Failed`.

**Caching**: the catalog listing (`getAvailableVehicles`) is cached for 5 minutes at the edge, with immediate invalidation on `VehicleSold`, `VehiclePublished`, `VehicleReserved`, `VehicleReleased`, and `VehicleCheckoutStarted` — the last two close a gap found during reconciliation: a vehicle released from a cancelled reservation, or one that just entered checkout, was not previously busting the cache in either direction.

## 9. Reservation Service

File: `modules/reservation/api/reservation-service.cds`

**Exposed Entities**: `Reservations`.

**Actions**:

* `createReservation` — `{ vehicleId, reservationDate }`. **Guest-accessible** (no `@requires: 'authenticated-user'` on this action specifically — see Section 18 for how it's still rate-limited and abuse-protected). Returns the created `Reservation` plus a `guestToken` when the caller is unauthenticated.
* `claimReservation` *(new)* — `{ reservationId, guestToken }`, called once the guest authenticates or registers at checkout. Emits `ReservationClaimed`, sets `customer_ID`, clears `guestToken`. Requires an authenticated session to call.
* `approveReservation` / `rejectReservation` / `completeReservation` — Allowed Roles: Admin, Manager, Operator (branch-scoped per Authorization Matrix Section 6).
* `cancelReservation` — Allowed Roles: Customer (Own, or guest presenting `guestToken`), Admin, Manager, Operator.

**Business rule enforced here, not just documented**: `createReservation` rejects with a domain exception if an `ACTIVE` reservation already exists for the vehicle (backed by the CDS partial unique index) — the handler must not assume the guard check alone is sufficient under concurrent requests.

## 10. Test Drive Service

File: `modules/test-drive/api/test-drive-service.cds`

**Exposed Entities**: `TestDrives`.

**Actions**:

* `requestTestDrive` — `{ vehicleId, date, time, contactEmail, contactPhone }`. **Guest-accessible**; `contactEmail`/`contactPhone` are required when unauthenticated, since a Test Drive carries no later claim step (Domain Model Rule TD-5).
* `approveTestDrive` / `completeTestDrive` — Allowed Roles: Admin, Manager, Operator.
* `cancelTestDrive` — Allowed Roles: Customer (Own), Admin, Manager, Operator.

**Functions**: `getAvailableTimeSlots(vehicleId)` — enforces Domain Model Rule TD-3 (no two test drives in the same slot for the same vehicle).

---

## 11. Offer Service

File: `modules/offer/api/offer-service.cds`

**Exposed Entities**: `Offers`.

**Actions**:

* `submitOffer` — `{ vehicleId, offeredPrice, desiredPickupDate }`. **Requires authentication** — unlike Reservation/Test Drive, there is no guest path here.
* `approveOffer` — Allowed Roles: Admin, Manager. Business result: `Offer → Approved → Vehicle Reserved` (the Reservation it creates inherits the Offer's `customer_ID` directly, never a `guestToken`).
* `rejectOffer` — Allowed Roles: Admin, Manager.
* `expireOffer` — System action, fired by the same expiry poller that handles Reservation expiry (48-hour window).

## 12. Sales Service

File: `modules/sales/api/sales-service.cds`

**Exposed Entities**: `Orders`.

**Actions**:

* `createOrder` — `{ vehicleId, deliveryType }`. Rejects if an active Order already exists for the vehicle (CDS Data Model Section 12 unique index).
* `cancelOrder` — Allowed Roles: Customer, Admin, Manager.
* `completeOrder` — Allowed Roles: Admin, Manager.

**Functions**: `getOrderDetails(orderId)`.

---

## 13. Payment Service

File: `modules/payment/api/payment-service.cds`

**Exposed Entities**: `Payments`.

**Actions**:

* `initiatePayment` — `{ orderId, provider }`. Requires a client-generated `idempotencyKey` in the request — the handler rejects a request with no key rather than silently generating one, so a retried request is provably the same request.
* `retryPayment` — Customer only.
* `refundPayment` — Allowed Roles: Admin, Manager.

**Functions**: `getPaymentStatus(orderId)`.

This service **never accepts or returns raw card data** — `initiatePayment` returns a PSP redirect/session reference, not a form for card entry; AutoMarket's servers stay out of PCI-DSS scope beyond SAQ-A (AD-4).

## 15. Notification Service

File: `modules/notification/api/notification-service.cds`

**Exposed Entities**: `Notifications` (read only).

**Functions**: `getMyNotifications`, `getUnreadCount`.

This service is the consumer side of the Favorites-driven triggers (`VehiclePriceDropped`, `VehicleSold`, `SimilarVehicleListed`) — it has no actions to *create* a notification directly; all notifications originate from domain events via the outbox.

## 16. Reporting Service

File: `modules/reporting/api/reporting-service.cds`

**Exposed Entities**: `VehicleSalesReport`, `ReservationReport`, `OfferConversionReport`, `BranchPerformanceReport`.

**Functions**: `getSalesDashboard`, `getBranchPerformance`, `getConversionRates(funnelType)` — `funnelType` is `'direct'` or `'reservation-led'`; the two funnels are reported separately, never merged, so a drop in reservation-led conversion isn't hidden by a healthy direct-purchase number.

**Authorization**: Admin, Manager.

## 18. Rate Limiting & API Versioning

**Rate limiting** is enforced at the Approuter, not per-service — see the Authorization Matrix, Section 18, for the full table. The distinction that matters here: `createReservation` and `requestTestDrive` are the two actions in this catalog reachable **without** authentication, and they fall under the stricter guest write limit (20 req/min per IP) rather than the authenticated-user limit.

**API Versioning**:

```text
Current Version: v1
Service Names: VehicleServiceV1, ReservationServiceV1, OfferServiceV1
Future versions: VehicleServiceV2 may coexist.
```

Event payload versioning follows the same principle conceptually (Solution Architecture Document): a breaking event payload change is a new event `type`, not a silent field change on the existing type, since Notification/Audit/Reporting all consume these without renegotiation.

## 20. Service Security

Authorization layers: XSUAA Roles → CAP `@restrict` → ABAC validation → Domain validation.

```text
Customer → Own Reservation       → Allowed
Customer → Other Customer's Reservation → Denied
Guest    → Reservation matching presented guestToken → Allowed
Guest    → Any other Reservation → Denied
```

## 21. Technical Rules

| Rule | Statement |
|---|---|
| API-1 | No service exposes database entities directly. |
| API-2 | All write operations must be actions. |
| API-3 | Business logic must never be implemented in CAP handlers. |
| API-4 | CAP handlers only orchestrate Use Cases. |
| API-5 | All services must be independently testable. |
| API-6 *(new)* | Any action reachable without authentication (`createReservation`, `requestTestDrive`) must validate a rate-limit and an abuse signal before touching the domain layer — guest write paths are exactly where AD-39 was added because this had no enforcement point originally. |
