## 1. Purpose

This document defines the Fiori application landscape of the AutoMarket platform: applications, user journeys, screen hierarchy, navigation flows, application ownership, and service consumption.

---

## 2. Application Landscape

```text
Customer Portal
Operator Portal
Manager Portal
Admin Portal
```

All four are compiled SAPUI5/Fiori apps hosted in the **HTML5 Application Repository**, served through the **Application Router**, which terminates session/token handling before any app code runs. No app is reachable by any other path.

## 3. Customer Portal

Application ID: `customer-portal`

Target audience: **Customer and Guest** — this portal is the one place in the landscape that serves unauthenticated traffic, per Guest Checkout.

Purpose: allow customers (and guests, where permitted) to browse, reserve, test drive, purchase vehicles, and manage their activities.

### Navigation Structure

```text
Home
├── Vehicle Catalog
├── Vehicle Details
├── Favorites           (requires login)
├── Reservations
├── Test Drives
├── Offers              (requires login)
├── Orders
├── Payments
└── Profile             (requires login)
```

### Guest Access Banner *(new)*

Every screen reachable pre-login (Vehicle Catalog, Vehicle Details, Reservation creation, Test Drive request) shows a persistent banner once a guest reservation exists: *"You have an active reservation — sign in or register to complete your purchase."* This is the UI surface for the `claimReservation` action (OData Service Design Document, Section 9) — it is not optional decoration, it's how a guest actually discovers they need to claim before checkout.

### Home Page

Widgets: Featured Vehicles, Recently Added Vehicles, Recommended Vehicles, Reservation Status Summary, Active Offers Summary (Active Offers Summary only renders post-login, since Offer has no guest path).

Consumed Services: `VehicleService`, `ReservationService`, `OfferService`.

### Vehicle Catalog

Purpose: browse available vehicles. Available to guests.

Functions: Search, Filter, Sort, Pagination.

Filters: Brand, Model, Year, Price Range, Fuel Type, Transmission, Mileage, Branch.

Consumed Service: `VehicleService` (results may be served from the 5-minute edge cache — see OData Service Design Document, Section 6).

### Vehicle Details

Sections: Images, Specifications, Price (with price-history sparkline via `PricingService`), Vehicle History, Branch Information.

Actions:

* Add to Favorites *(requires login)*
* Reserve Vehicle *(available to guests — creates a `guestToken`-backed reservation if not logged in)*
* Request Test Drive *(available to guests — requires contact email/phone if not logged in)*
* Submit Offer *(requires login)*
* Buy Vehicle *(requires login or triggers the claim flow if a guest reservation exists)*

Consumed Services: `VehicleService`, `PricingService`, `FavoriteService`, `ReservationService`, `OfferService`, `SalesService`.

### Favorites

Purpose: manage favorite vehicles. **Requires login** — not part of Guest Checkout, since a Favorite is the subscription target for re-engagement notifications and a guest has no durable address to notify.

Actions: Add Favorite, Remove Favorite, Navigate to Vehicle.

Consumed Service: `FavoriteService`.

### Reservations

Purpose: manage reservations — works for both guests (scoped to their `guestToken`) and logged-in customers.

Actions: Create Reservation, Cancel Reservation, View Reservation Status, **Claim Reservation** *(new — surfaces at checkout for a guest-created reservation)*.

Consumed Service: `ReservationService`.

### Test Drives

Purpose: manage test drive requests — works for guests via contact-detail matching, no claim step needed.

Actions: Request Test Drive, Cancel Test Drive, View Schedule.

Consumed Service: `TestDriveService`.

### Offers

Purpose: manage submitted offers. **Requires login.**

Actions: Submit Offer, View Offer Status.

Consumed Service: `OfferService`.

### Orders

Purpose: track purchases. Actions: View Orders, Cancel Order.

Consumed Service: `SalesService`.

### Payments

Purpose: manage payment transactions. Actions: Initiate Payment, Retry Payment, View Payment Status.

Consumed Service: `PaymentService`. This screen never renders a raw card-number field — payment capture happens on the PSP's hosted page/redirect (AD-4).

### Profile

Purpose: manage personal information. **Requires login** (a guest has no profile to manage). Actions: Update Profile, Change Password.

Consumed Service: `IdentityService`.

## 4. Operator Portal

Application ID: `operator-portal`

Target Role: Operator. Purpose: manage day-to-day dealership operations, scoped to the Operator's assigned branch.

### Navigation Structure

```text
Dashboard
├── Vehicles
├── Reservations
├── Test Drives
├── Offers
├── Customers
└── Notifications
```

### Dashboard

Widgets: Vehicles For Sale, Active Reservations, Upcoming Test Drives, Pending Customer Requests — all filtered to the Operator's branch (ABAC, Authorization Matrix Section 6/7).

Consumed Services: `VehicleService`, `ReservationService`, `TestDriveService`.

### Vehicles

Actions: Create Vehicle, Update Vehicle, Publish Vehicle, Change Status. Consumed Service: `VehicleService`.

### Reservations

Actions: Approve Reservation, Reject Reservation, Complete Reservation. Includes both customer- and guest-originated reservations (an Operator does not need a customer identity to act on a reservation — the branch-scoped ABAC rule evaluates the denormalized `branch` column regardless). Consumed Service: `ReservationService`.

### Test Drives

Actions: Approve Test Drive, Cancel Test Drive, Complete Test Drive. Consumed Service: `TestDriveService`.

### Offers

Purpose: view offers submitted by customers. Actions: View Offer. **Note: Operators cannot approve offers.** Consumed Service: `OfferService`.

## 5. Manager Portal

Application ID: `manager-portal`

Target Role: Manager. Purpose: manage sales operations and branch performance.

### Navigation Structure

```text
Dashboard
├── Vehicles
├── Reservations
├── Offers
├── Orders
├── Deliveries
└── Reports
```

### Dashboard

Widgets: Monthly Sales, Reservation Conversion, Offer Conversion, Branch Performance. Consumed Service: `ReportingService`.

### Offers

Actions: Approve Offer, Reject Offer, Expire Offer. Approving creates a Reservation that inherits the Offer's identity directly (Domain Model Rule O-2). Consumed Service: `OfferService`.

### Orders

Actions: View Orders, Complete Orders, Cancel Orders. Consumed Service: `SalesService`.

### Deliveries

Actions: Schedule Delivery, Update Delivery, Complete Delivery. Consumed Service: `DeliveryService`.

### Reports

Available Reports: Vehicle Sales Report, Reservation Report, Offer Conversion Report, Branch Performance Report. **Two conversion views, not one**: direct-purchase funnel and reservation-led funnel are shown separately (Domain Model Section 18) — a single blended conversion number would hide whether one path is underperforming the other. Consumed Service: `ReportingService`.

## 6. Admin Portal

Application ID: `admin-portal`

Target Role: Admin. Purpose: system administration and governance.

### Navigation Structure

```text
Dashboard
├── Users
├── Roles
├── Branches
├── Audit Logs
├── Notifications
└── System Configuration
```

### Users

Actions: Create User, Disable User, Assign Role. Consumed Services: `AdminService`, `IdentityService`. The user form surfaces the MFA-required flag (read-only, derived from role — Operator/Manager/Admin are always MFA-mandatory) and lockout status.

### Roles

Actions: View Role Assignments, Manage Role Collections. Consumed Service: `AdminService`.

### Branches

Actions: Create Branch, Update Branch, Disable Branch. The branch form includes the **Region** field (data residency) added to the Domain Model. Consumed Service: `BranchService`.

### Audit Logs

Actions: Search Audit Logs, Export Audit Logs. Read-only, 7-year retention. Consumed Service: `AdminService`.

### Notifications

Actions: View Notification Queue, Resend Notification. Consumed Service: `NotificationService`.

## 7. Cross Application Navigation

### Customer Portal

```text
Vehicle Catalog → Vehicle Detail → Buy Vehicle → Create Order → Payment
Vehicle Detail → Reserve Vehicle → Reservation Confirmation
                                  → (if guest) Claim Reservation banner → Login/Register → Checkout
Vehicle Detail → Request Test Drive → Schedule Selection
```

### Operator Portal

```text
Reservation → Approve → Vehicle Reserved
```

### Manager Portal

```text
Offer → Approve → Reservation Created
```

## 8. Fiori Design Principles

* Role-Based Applications
* Responsive Design
* Mobile Compatibility
* Accessibility Compliance
* Consistent Navigation
* Simple User Experience
* **Guest-first where permitted** *(new)* — a screen that Guest Checkout allows pre-authentication must never force a login wall; the claim/login prompt appears only at the point Guest Checkout actually requires it (checkout).
