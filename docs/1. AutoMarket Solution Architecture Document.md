# Document Control

This document **supersedes v1** for all architecture decisions below. It does not repeat business story, customer journeys, or functional requirements already defined in `AutoMarket-Technical-Documentation-v1.md` — it resolves the gaps and errors identified in the v1 architecture review:

* Vehicle and Payment state machines were drawn as strictly linear, which contradicted the textual scenarios (cancellations, rejections, direct-purchase skipping RESERVED).
* No bounded context map, no monolith-vs-microservices decision, no multi-tenancy decision.
* No concurrency strategy for the "only one active reservation per vehicle" rule.
* No event delivery guarantees (outbox/idempotency) despite an EventBus being a Phase 1 core service.
* No PCI-DSS posture despite a Payment domain.
* "ABAC" named without an implementation mechanism.
* Missing Delivery and Pricing domains referenced by the customer journey but absent from the domain model.
* No DR/RPO/RTO targets, no IaC strategy, no test strategy beyond a principle statement.

Each item is resolved below as an explicit, recorded decision so engineering does not have to re-litigate it during implementation.

**Revision note**: this version also incorporates a second, multi-disciplinary review pass (Enterprise/BTP/CAP/DDD/Security/Cloud/DevOps/QA/Data/Performance/SRE/Product perspectives), closing 10 further gaps: Application Router & HTML5 App Repository topology, SLO targets, schema migration policy, audit/event retention, MFA enforcement, account lockout, a Reporting/Analytics domain, concrete cache TTLs with invalidation, and explicit DDD aggregate boundaries. These are recorded as AD-9 through AD-17 below.

**Revision note 2**: a third review pass scored 15 architecture categories and found this version still short of "implementation-ready" in 15 specific, named ways — no business capability map/TCO awareness, no saga/choreography decision, no API/event versioning, no BTP subaccount topology, no network segmentation, no threat model/DPIA/rate-limit enforcement, no deployment cadence/coding standards, no coverage threshold/chaos testing, no data classification/migration plan, no latency budget/capacity model, no error budget/runbook, no funnel instrumentation, no governance sign-off gate. All are closed below as AD-19 through AD-36, plus a new Business Architecture section and an Ubiquitous Language glossary.

**Revision note 3**: a fourth pass re-checked this document's *internal* consistency (not just coverage) and found five real cross-section conflicts: the Saga section falsely claimed Offer approval was the only multi-aggregate workflow (Checkout already is one); Reporting/Analytics had no retention story independent of the 2-year Domain Event window; the migration plan conflated marketing consent with transactional data's separate legal basis; the latency budget didn't say whether it applied before or after a cache hit; and the ABAC CDS example had no denormalized `branch_ID` to actually evaluate against. Closed as AD-37, AD-38, and targeted fixes to the Saga, Reporting, Migration, and Performance Budget sections. The same pass also caught a naming drift (`PaymentCaptured` vs. the canonical `PaymentSucceeded` from the v1 Event Catalog) in two places and corrected it.

**Revision note 4**: a fifth pass closed the Funnel Instrumentation gap (it covered only the reservation-led path) by splitting it into the direct-purchase and reservation-led funnels, tracked separately so v1's "improve reservation management" objective is actually measurable. The same pass found Guest Checkout's downstream effects were never threaded through two other sections: Rate Limiting (AD-24) is scoped to "authenticated user" only, leaving guest-created Reservations/Test Drives — which Guest Checkout explicitly permits — outside any rate limit; and the Concurrency/ABAC model assumes a Reservation always has a `customer_ID`, with no defined flow for converting a guest's anonymous reservation into an identified Customer record at the payment step.

**Revision note 5**: both findings from the fifth pass are now closed — AD-39 (guest-scoped, IP-based rate limits) and AD-40 (`guestToken` + `ReservationClaimed`, scoped correctly to Reservation only since Offer already requires authentication). A sixth pass, prompted by writing AD-40, found that "reservation still within validity window" is used as a guard condition twice in the Vehicle State Machine but its actual duration is never specified anywhere in v1 or v2 — an open finding, not yet closed.

**Revision note 6**: the sixth pass's finding is now closed as AD-41 (48-hour default, anchored to the original timestamp, flagged pending product confirmation in Open Decisions), plus an expiry-job mechanism added to the Concurrency section. Writing AD-41 prompted a seventh pass over the Caching section (AD-16): the cache-invalidation event list (`VehicleSold`, `VehiclePublished`, `VehicleReserved`) never included `VehicleReleased` — the event that fires when `ReservationCancelled`/`ReservationExpired`/`PaymentFailed` return a vehicle to `FOR_SALE` — meaning a just-released vehicle could show as unavailable in the cached catalog for up to 5 minutes. A related, smaller finding: no Vehicle event in the v1 Event Catalog represents entering `PENDING_PAYMENT` at all, so that transition can't bust the cache either.

**Revision note 7**: both seventh-pass findings are closed — `VehicleCheckoutStarted` is added to the Vehicle Event Catalog (AD-42), and the cache-bust list now includes it alongside `VehicleReleased`. An eighth pass swept the previously stable sections (Subaccount Topology, Threat Model, Governance & Sign-Off, Data Classification) and found no issue of comparable severity to the prior seven — only a minor, optional note: the Data Classification table (AD-30) classifies catalog data, branch data, customer PII, and payment tokens, but never explicitly classifies Domain Events themselves, which can carry a `customer_ID` in their payload. Left open as a minor item, not a scored finding, since it's a classification-completeness gap rather than a missing control.

# Business Architecture

v1/v2 covered customer journeys and NFRs but produced no business-architecture artifacts. Closing that gap:

## Business Capability Map

| L1 Capability | Maps to Domain(s) |
|---|---|
| Customer Engagement | Search, Favorites / Notification |
| Sales Execution | Reservation, Offer, Test Drive, Sales |
| Fulfillment | Payment, Delivery |
| Inventory & Pricing | Vehicle, Pricing |
| Operations Management | Branch, Reporting/Analytics |
| Platform Foundations | Identity, Audit |

## Value Streams

* **Lead-to-Delivery** (primary revenue stream): Discover (Search/Catalog) → Engage (Favorite / Reserve / Offer / Test Drive) → Transact (Payment) → Fulfill (Delivery) → Retain (Favorites-driven re-engagement, AD-18).
* **Inventory-to-Listing** (supply-side stream): Intake → Price (Pricing Domain) → Publish (`VehiclePublished`) → Monitor availability → Retire (`VehicleSold` / `VehicleDelivered`).

Every domain in the Bounded Context Map exists to serve exactly one step in one of these two streams — a domain that doesn't map to either is a candidate for removal, not addition.

## Cost Model (TCO Awareness)

Exact pricing requires an SAP account team quote and is deliberately not fabricated here. What an architecture document *can* responsibly state is the cost drivers, so procurement asks the right questions:

| Cost driver | Scales with | Phase |
|---|---|---|
| HANA Cloud compute/storage | Vehicle/customer/event volume | 1 — right-sized per the Scalability section |
| Cloud Foundry runtime (memory/instances) | Concurrent request load | 1 |
| XSUAA / IAS | Active user count (often banded pricing) | 1 |
| Application Autoscaler | Peak-to-average load ratio | 2 |
| Event Mesh | Message volume | 3 (AD-5) |
| SAP Datasphere (if adopted) | Reporting data volume/complexity | 4+, only if AD-15 is revisited |

This table is the input to a procurement conversation, not a substitute for one — see Open Decisions.

# New Domains (closing the v1 gap)

## Delivery Domain

Referenced in the customer journey ("Selects a delivery option") but absent from v1's domain list.

Responsibilities:

* Delivery method selection (Home Delivery / Pick-Up)
* Delivery address validation
* Delivery scheduling and status tracking (`SCHEDULED → IN_TRANSIT → DELIVERED`)

Relationship: an `Order` (Sales Domain) has exactly one `Delivery`. `VehicleDelivered` event (already in the v1 Event Catalog) is emitted by this domain, not by Vehicle.

## Pricing Domain

Referenced implicitly (offers are "too high," Phase 4 mentions "Vehicle Pricing Intelligence") but no Phase 1 pricing model existed.

Responsibilities (Phase 1 — intentionally minimal):

* List price per vehicle, currency, tax handling
* Offer-vs-list-price comparison used by Offer Domain during review
* Price history (audit trail for changes, feeds Phase 4 pricing intelligence later)

This is deliberately thin in Phase 1 — it exists so Phase 4's "Pricing Intelligence" has a domain to extend rather than retrofitting one later.

## Branch Domain (formalized)

v1 mentioned "Branches" only as a Master Data entity. Given ABAC depends entirely on branch scoping (AD-3), it is promoted to a first-class domain:

```text
Branch
 ├─ ID
 ├─ Name
 ├─ Address
 ├─ ManagerID
 └─ Region            (added — needed for data residency, see Enterprise Readiness)
```

Every `Vehicle`, `Operator`, and `Manager` belongs to exactly one `Branch`. `Customer` does not belong to a branch (customers may transact with any branch).

## Reporting / Analytics Domain

v1's "Manager: Review reports" requirement implied reports would be queried directly off the operational (OLTP) tables — Vehicle, Sales, Offers, Reservations. That couples reporting load to the transactional hot path and was never called out as a separate concern.

Responsibilities:

* A reporting schema, populated **asynchronously** by the same outbox event stream that already feeds Notification and Audit (AD-15) — no new infrastructure, just another consumer.
* HANA Cloud calculation views over that reporting schema for Manager dashboards (sales velocity, reservation conversion, offer approval rate).
* This is the **Analytical Model**, kept structurally separate from the **Operational Model** (the per-domain CAP services). The two must never share a query path.
* **Retention is independent of the raw event stream** (AD-37): Domain Events expire per AD-12 (2 years), but the reporting schema's own aggregates are not deleted on that schedule. Purging raw events at 2 years must never silently break a Manager's year-over-year comparison.

This is intentionally Phase 1 scope at minimal cost — it reuses the outbox rather than introducing a data warehouse. Moving to SAP Datasphere is a Phase 4+ option if reporting needs outgrow calculation views (see Open Decisions).

# Aggregate Boundaries (DDD)

v1/v2 named domains but never stated where one transactional consistency boundary ends and another begins. This matters concretely because the Concurrency & Consistency Strategy (below) already locks the `Vehicle` row during state transitions — if `Offer`, `Reservation`, and `TestDrive` were modeled as children inside the `Vehicle` aggregate, every offer submission or test-drive booking would contend for that same lock.

Per AD-17:

* **Vehicle**, **Reservation**, **Offer**, **TestDrive**, **Order**, and **Payment** are each their own Aggregate Root with their own lifecycle and invariants.
* They reference each other **by ID only** (`vehicle_ID`, `reservation_ID`, …) — never as nested/embedded collections.
* `Vehicle` is deliberately kept small: it owns only its own state machine (DRAFT…DELIVERED) and its own fields. It does **not** own a collection of Offers or Reservations.
* Cross-aggregate invariants (e.g. "only one active reservation per vehicle") are enforced the way the Concurrency section already describes — a guard check plus a DB constraint — not by aggregate nesting.

This keeps each aggregate's transaction small and lets Offer/Reservation/TestDrive lifecycles change independently without taking a lock on Vehicle's full object graph.

## Cross-Aggregate Workflow Coordination (Saga)

Per AD-19, multi-aggregate workflows are coordinated by **choreography** — each aggregate reacts to events from others; there is no central orchestrator. Two workflows already require this (an earlier draft of this section incorrectly claimed Offer approval was "the only one," which contradicted the Vehicle State Machine's own `PaymentFailed` row below):

**Offer approval**:
1. `OfferApproved` is emitted by the Offer aggregate.
2. The Reservation aggregate consumes it and attempts `ReservationCreated`.
3. If step 2 fails (e.g. the vehicle was reserved by someone else in the race window), the Reservation aggregate emits `ReservationCreationFailed`.
4. The Offer aggregate consumes that and reverts itself to `PENDING` with an `OfferRequiresReReview` flag; the Operator is notified — no silent inconsistency between "Offer says approved" and "no Reservation exists."

**Checkout** (already implied by the Vehicle State Machine's `PENDING_PAYMENT` row, made explicit here):
1. `CheckoutStarted` moves Vehicle to `PENDING_PAYMENT` and creates an Order (Sales aggregate).
2. The Payment aggregate processes `INITIATED → AUTHORIZED → CAPTURED` (or `FAILED` at either step, per the Payment State Machine).
3. On `PaymentSucceeded`, Vehicle consumes it and transitions to `SOLD`; Order consumes it and completes.
4. On `PaymentFailed`, Vehicle consumes it and reverts to `FOR_SALE` or `RESERVED` per the Vehicle State Machine table, and the Order is cancelled.

Both are choreography today, not a saga engine, because each has only one compensating step. The threshold to watch is AD-19's revisit trigger (2-3 compensating steps) — not "the first workflow to appear," which is the claim this section corrects.

## Guest Reservation Identity (AD-40)

Guest Checkout (Enterprise Readiness) permits an unauthenticated customer to create a **Reservation** — note: not an Offer, which already requires authentication and is absent from the Guest Checkout allow-list — but the Vehicle State Machine's ownership guard and the CDS ABAC example both assume a `customer_ID` already exists. This section closes that gap:

* A guest-created Reservation carries a signed `guestToken` (returned to the browser, not a login) instead of a `customer_ID`.
* The guest can read or cancel their own reservation by presenting that token — no XSUAA/IAS session required for that alone.
* At `CheckoutStarted`, full identity is required (per Guest Checkout policy): the guest authenticates or registers, which emits `ReservationClaimed` and sets `customer_ID` on the reservation.
* From that point, the existing ownership guard ("requester is the reservation/offer owner") and the ABAC `where: customer_ID = $user` check apply exactly as already specified — nothing downstream of the claim needed to change.

This does not extend to Test Drive requests, which never carry an ownership guard later in their lifecycle (a sold vehicle simply cancels the test drive and notifies whatever contact info was captured at booking) — only Reservation needed this, because only it has a later step that checks "is this the same person."

| Term | Definition |
|---|---|
| Vehicle | A sellable inventory unit tracked through the Vehicle State Machine, owned by exactly one Branch |
| Active Reservation | A Reservation in a non-terminal status — the target of the one-active-reservation-per-vehicle uniqueness constraint |

# Corrected State Machines

The v1 diagrams were drawn as straight top-to-bottom arrows, which is incorrect: it implied a single forward-only path with no cancellations, rejections, or direct-purchase shortcuts. Below are the actual transition tables — these are authoritative; do not redraw them as a linear chain.

## Vehicle State Machine

| From | To | Trigger | Guard |
|---|---|---|---|
| DRAFT | FOR_SALE | `VehiclePublished` | required fields complete (price, images, branch) |
| FOR_SALE | RESERVED | `ReservationCreated` | no active reservation exists for the vehicle (AD-6) |
| FOR_SALE | RESERVED | `OfferApproved` | same guard as above |
| FOR_SALE | PENDING_PAYMENT | `CheckoutStarted` (direct purchase, skips reservation) | vehicle not currently RESERVED by another customer |
| RESERVED | PENDING_PAYMENT | `CheckoutStarted` | requester is the reservation/offer owner — for a guest-created reservation this is where `ReservationClaimed` assigns a `customer_ID` (AD-40); reservation still within its 48-hour validity window (AD-41) |
| RESERVED | FOR_SALE | `ReservationCancelled` / `ReservationExpired` / inspection declined | — |
| PENDING_PAYMENT | SOLD | `PaymentSucceeded` | — |
| PENDING_PAYMENT | FOR_SALE | `PaymentFailed` (originated from direct purchase) | — |
| PENDING_PAYMENT | RESERVED | `PaymentFailed` (originated from a reservation/offer) | reservation still within its 48-hour validity window (AD-41) |
| SOLD | DELIVERED | `DeliveryConfirmed` | — |

`RESERVED` and `PENDING_PAYMENT` are **not** mandatory waypoints — direct purchase (Scenario 1) legitimately skips `RESERVED`. This was implied by v1's text but contradicted by its diagram; the table above is the single source of truth.

**Event Catalog extension (AD-42)**: every other Vehicle status has a corresponding emitted event (`VehiclePublished`, `VehicleReserved`, `VehicleReleased`, `VehicleSold`, `VehicleDelivered`) except entry into `PENDING_PAYMENT` — v1's Event Catalog simply never defined one. `VehicleCheckoutStarted` is added to close that gap; Vehicle emits it on both `CheckoutStarted` rows above (from `FOR_SALE` or from `RESERVED`).

## Payment State Machine

| From | To | Trigger |
|---|---|---|
| INITIATED | AUTHORIZED | PSP authorization succeeded |
| INITIATED | FAILED | PSP authorization declined or timed out |
| AUTHORIZED | CAPTURED | Capture succeeded on order confirmation |
| AUTHORIZED | FAILED | Capture failed or authorization expired |
| CAPTURED | REFUNDED | Refund issued (full or partial) |

`FAILED` and `REFUNDED` are **separate terminal branches**, not a continuation of `CAPTURED`. A captured (successful) payment cannot become `FAILED`; a failed payment cannot be `REFUNDED` (there is nothing to refund). The v1 diagram chained all five states sequentially, which is not a valid payment lifecycle.

# Concurrency & Consistency Strategy

This was entirely unspecified in v1 despite an explicit business rule ("No other customer can reserve the vehicle").

* **Optimistic locking** on `Vehicle` via CAP's built-in `@odata.etag` — every update carries a version check; conflicting concurrent updates are rejected at the HTTP layer (`412 Precondition Failed`).
* **Transition guard inside the transaction**: the state-transition handler issues `SELECT ... FOR UPDATE` on the vehicle row before evaluating the guard in the table above, closing the read-check-write race window (AD-6). This is a short-lived row lock, not application-level pessimistic locking.
* **One active reservation per vehicle**: enforced by a partial unique constraint (`UNIQUE (vehicle_ID) WHERE status = 'ACTIVE'`) in addition to the application-level guard — defense in depth against bugs in the guard logic.
* **Payment idempotency**: every checkout attempt carries a client-generated idempotency key, stored unique per order, so PSP retries (network timeouts, double-clicks) cannot result in a double capture.
* **Reservation expiry** (AD-41): a periodic job — the same pattern as the outbox dispatch poller — scans `ACTIVE` reservations past `createdAt + 48h` and fires `ReservationExpired`. The window is anchored to the original `ReservationCreated`/`OfferApproved` timestamp; a failed checkout attempt does not reset it, so a customer cannot indefinitely tie up a vehicle by repeatedly attempting and failing checkout near the deadline.

# Event-Driven Backbone

v1 listed an `EventBus` as a Phase 1 core service but deferred SAP Event Mesh to Phase 3, leaving an undefined gap: what guarantees does the Phase 1 EventBus actually provide?

* **Transactional outbox (Phase 1)**: domain events are written to the `DomainEvents` table in the *same database transaction* as the business state change. A lightweight in-process poller dispatches unprocessed rows to subscribers (NotificationQueue, AuditService, Search index updater).
* **At-least-once delivery, idempotent consumers**: every consumer is required to be idempotent (dedup by event ID) because outbox dispatch guarantees at-least-once, not exactly-once.
* **Event envelope** (consistent shape for every event in the v1 Event Catalog):

```json
{
  "eventId": "uuid",
  "type": "VehicleReserved",
  "occurredAt": "ISO-8601",
  "aggregateId": "vehicle-id",
  "branchId": "branch-id",
  "payload": { }
}
```

* **Phase 3 migration**: only the *dispatch* step changes (publish to Event Mesh instead of in-process invocation). Consumers are unaffected because they already assume at-least-once + idempotency — this is the point of building it this way now rather than retrofitting reliability later.
* **Contract tests**: each event type's payload schema is validated in CI against a JSON Schema per event, so a producer change that breaks a consumer is caught before merge.

# Favorites-Driven Re-Engagement Notifications

v1 listed `Favorites` only as a Business Data entity with no behavior attached. Per AD-18, this is pulled into scope now rather than left as an unscoped later-phase idea, because the architecture above already has everything needed to support it cheaply.

New events (added to the Event Catalog, dispatched through the existing outbox — no new infrastructure required):

* `VehiclePriceDropped` — emitted by the Pricing Domain when a vehicle's list price decreases
* `VehicleSold` — already exists (v1 Event Catalog), reused as-is
* `SimilarVehicleListed` — emitted by the Vehicle Domain on `VehiclePublished`, when the new vehicle's attributes (make, model, category, branch) match an existing Favorite

**Consumer**: the Notification Domain subscribes to all three. For each event it resolves the customers who favorited the relevant vehicle (or, for `SimilarVehicleListed`, the customers who favorited a matching vehicle) and queues one notification per favorite — the same `NotificationQueue` mechanism already defined for offer/reservation notifications.

**Scope boundary** (intentional, to stop this from growing into a recommendation engine): "similar" means attribute equality (make + model + category + branch), not fuzzy or learned-preference matching. A preference-based matching engine belongs to Phase 4 ("AI-Based Recommendations"), not here — conflating the two would turn a cheap notification feature into a much larger one.

ABAC implementation (AD-3) using CAP-native annotations — no external AMS dependency:

```cds
@requires: 'authenticated-user'
service ReservationService {

  @restrict: [
    { grant: 'READ',      to: 'Customer', where: 'customer_ID = $user' },
    { grant: '*',         to: 'Operator', where: 'branch_ID = $user.branch' },
    { grant: '*',         to: 'Manager',  where: 'branch_ID = $user.branch' },

}
```

`$user.branch` is populated from the XSUAA/IAS token attribute at login — this is the entire ABAC mechanism. If attribute rules grow beyond branch scoping, AD-3 should be revisited in favor of SAP AMS rather than accumulating ad-hoc `where` clauses.

**Multi-tenancy note**: per AD-2, CAP MTX is intentionally *not* part of Phase 1. If the business later decides to resell AutoMarket to independent dealer groups (true SaaS), MTX onboarding is a scoped follow-on project, not a retrofit of the data model — branch scoping already isolates data by `branch_ID`, which is one prerequisite MTX would need anyway.

## Example Entity Definition

The module layout above is abstract; a concrete `db/vehicle/schema.cds` looks like this (trimmed to the fields the Concurrency and State Machine sections already depend on):

```cds
namespace automarket.vehicle;
using { automarket.master.Branch } from '../master/schema';

entity Vehicles {
  key ID        : UUID;
  branch        : Association to Branch;
  make          : String(50);
  model         : String(50);
  category      : String(30);
  status        : String(20) enum {
    DRAFT; FOR_SALE; RESERVED; PENDING_PAYMENT; SOLD; DELIVERED;
  };
  listPrice     : Decimal(10,2);
  currency      : Currency;
  modifiedAt    : Timestamp @cds.on.update : $now;  // backs @odata.etag, see AD-6
}
```

`status` is constrained to the Vehicle State Machine's states; the transition guard logic (the `FOR UPDATE` + table-driven checks from the Concurrency section) lives in the service handler, not the CDS model — CDS declares shape and constraints, not behavior.

Reservation, Offer, and TestDrive denormalize `branch_ID` from their Vehicle at creation time (AD-38), so the ABAC `@restrict` example above evaluates against a direct column instead of joining through Vehicle on every read:

```cds
entity Reservations {
  key ID        : UUID;
  vehicle       : Association to automarket.vehicle.Vehicles;
  branch_ID     : UUID;   // denormalized from vehicle.branch at creation — AD-38
  customer_ID   : UUID null;   // null until claimed — see Guest Reservation Identity (AD-40)
  guestToken    : String(64) null;   // set instead of customer_ID for guest-created reservations
  status        : String(20) enum { ACTIVE; CANCELLED; EXPIRED; COMPLETED; };
}
```

## API & Event Versioning (AD-20)

* OData services are versioned by URL path: `/odata/v4/vehicle/v1`. A breaking change gets a new path segment (`/v2`); additive changes (new optional field) don't bump the version.
* Event payloads (the envelope in the Event-Driven Backbone section) follow the same principle — a breaking payload change is a new event `type` value, not a silent field change on the existing type, since Notification/Audit/Search/Reporting all consume these without renegotiation.
* Deprecated versions get a 6-month sunset window, advertised via a `Sunset` HTTP header, before removal.

# Security & Compliance

## Threat Model Summary (STRIDE)

A quick pass across the main trust boundary (Browser → Approuter → CAP → HANA, plus inbound PSP webhooks):

| Threat | Mitigation | Where |
|---|---|---|
| Spoofing | XSUAA/IAS token validation terminates at the Approuter; nothing downstream trusts an unverified caller | AD-9 |
| Tampering | TLS everywhere, optimistic locking, webhook signature verification | AD-6, Payment Security |
| Repudiation | Audit Domain records who/what/when/why for every business action | Audit Domain |
| Information Disclosure | ABAC branch scoping, PCI SAQ-A scope reduction, data classification | AD-3, AD-4, AD-30 |
| Denial of Service | Rate limiting, Application Autoscaler (Phase 2) | AD-24 |
| Elevation of Privilege | Server-enforced `@restrict`; no client-supplied role is ever trusted | AD-3 |

This is a living checklist, not a one-time exercise — revisit it whenever a new external integration (a new PSP, a new inbound webhook) is added.

## Payment Security (closing the PCI-DSS gap)

* AutoMarket servers never receive, process, or store raw PAN/CVV (AD-4). Checkout uses PSP-hosted fields or redirect; AutoMarket stores only a PSP token + card last4 + brand.
* This keeps PCI-DSS scope at **SAQ-A**, avoiding a full Level 1 audit.
* Webhook signature verification required for all inbound PSP callbacks; replay protection via timestamp + nonce check.
* PSP vendor selection (Stripe / Adyen / SAP Multi-Bank Connectivity) is an **open decision for stakeholders** — see final section.

## RBAC / ABAC

* RBAC roles unchanged from v1: Customer, Operator, Manager, Admin.
* ABAC implemented as shown above (branch scoping via `@restrict` + `$user.branch`) — see AD-3.

## Authentication Hardening (closing the MFA / lockout gap)

v1 named MFA as a capability without stating when it applies. Per AD-13/AD-14:

| Control | Policy |
|---|---|
| MFA — Operator / Manager / Admin | **Mandatory** — internal roles can mutate inventory, approve offers, and access reports |
| MFA — Customer | Risk-based / optional at launch, to protect the Guest Checkout conversion goal | 
| Account lockout | 5 failed attempts within 15 minutes → 30-minute time-based lockout |
| Session idle timeout | Admin 15 min, Manager/Operator 30 min, Customer follows IAS default (enforced at the Approuter — see Production Topology) |
| Password hashing | Bcrypt (carried over from earlier drafts) |

## Rate Limiting (AD-24, AD-39)

| Scope | Limit | On breach |
|---|---|---|
| Authenticated user — write operations | 100 req/min | `429` + `Retry-After` |
| Authenticated user — read operations | 300 req/min | `429` + `Retry-After` |
| Anonymous/guest — write operations (Reservation, Test Drive creation) | 20 req/min per IP | `429` + `Retry-After` |
| Anonymous/guest — read operations | 100 req/min per IP | `429` + `Retry-After` |

Enforced at the Approuter/API layer, closing the v1 gap where "rate limiting" was named as an NFR with no enforcement point or limit attached. The guest rows (AD-39) close a second gap: AD-24 originally scoped limits to "authenticated user" only, leaving exactly the traffic Guest Checkout opens up (anonymous Reservation/Test Drive writes) unthrottled. Guest limits are intentionally stricter and IP-based, since anonymous traffic has no account to hold accountable.

## Secrets & Data Protection

* All credentials (PSP API keys, SMTP credentials) live in BTP Credential Store, never in environment variables or source control.
* Secret rotation cadence: 90 days, with a documented break-glass procedure for emergency rotation.

## Retention Policy (closing the audit/event retention gap)

| Data | Retention | Rationale |
|---|---|---|
| Audit Logs | 7 years | Legal/financial trail — **default pending legal confirmation per launch jurisdiction** (see Open Decisions) |
| Domain Events | 2 years | Operational replay/debugging value only, no legal retention requirement |
| Notification Logs | 1 year | Operational troubleshooting only |

GDPR's "right to be forgotten" is satisfied by pseudonymizing the customer reference on expiry/request, not by deleting the Audit Log row itself — the legal trail must survive, the personal identifier does not.

This table governs the **raw Domain Events stream** only. Reporting/Analytics projections derived from that stream are retained independently (AD-37) — they are not purged on the same 2-year schedule.

## GDPR

* Consent management, marketing opt-in, right to be forgotten, and data retention policies remain as defined in v1.
* **Data residency** (new): the BTP subaccount and HANA Cloud instance region must match the regulatory jurisdiction of the branches it serves (tracked via the new `Branch.Region` field). This must be confirmed before go-live in any new region.
* Lower environments (DEV/TEST/UAT) must use anonymized or synthetic data — no real customer PII outside PROD.

## Data Protection Impact Assessment (DPIA)

Per AD-25, a formal DPIA (GDPR Art. 35) is a **required pre-go-live gate**, not an optional best practice — Payment and identity data processing both meet the threshold that requires one. This document's Security & Compliance, Retention Policy, and Data Classification sections are the primary engineering input to that assessment, but the DPIA itself is owned and signed off by the Data Protection Officer, not engineering (see Governance & Sign-Off).

# Testability Strategy

v1 stated "Testability First" as a principle with no concrete plan. Target test pyramid:

| Layer | Target share | Tooling |
|---|---|---|
| Unit (domain logic, state machine, `VehicleAvailabilityService`) | ~70% | Jest |
| Integration (CAP service level, in-memory SQLite or HANA testcontainer) | ~20% | `cds test` |
| End-to-end (Fiori UI flows) | ~10% | Playwright / UI5 test tooling |

Additional requirements:

* **Contract tests** for every event type in the Event Catalog (schema validation in CI) — protects Notification, Audit, and Search consumers from silent breakage when a producer changes a payload.
* **State machine tests** must assert both valid transitions *and* that every invalid transition in the tables above is rejected.
* **Test data**: anonymized fixtures generated per environment refresh; no production PII copied into DEV/TEST/UAT.
* **Performance/load testing** (k6 or JMeter) against the `<500ms` API and `<2s` search NFRs and the SLO targets below — run against UAT before each PROD promotion, not just functional regression.
* **Code coverage** (AD-28): 80% minimum on domain/business-logic modules (state machine, guards, `VehicleAvailabilityService`), 60% overall — enforced as a CI merge gate, not just measured.
* **Chaos/fault-injection testing** (AD-29): periodically kill the outbox dispatch poller mid-batch in a non-prod environment to verify at-least-once delivery and consumer idempotency actually hold under failure — the Event-Driven Backbone's reliability guarantee has never been tested under failure until this exists.

# DevOps & Release Management

v1 defined environments (DEV/TEST/UAT/PROD) and GitHub Actions but no IaC, rollback, or promotion gates.

* **Infrastructure as Code**: BTP service instances and subscriptions defined in `mta.yaml`, reviewed via pull request like application code — no manually clicked service instances.
* **Promotion gates**:
  * DEV — auto-deploy on merge to `main`
  * TEST — auto-deploy + automated regression suite
  * UAT — manual gate, business sign-off required
  * PROD — manual approval + change ticket
* **Deployment strategy**: rolling deployment on Cloud Foundry by default. Blue-green deployment is deferred to Phase 2 (not needed at current traffic) but **rollback to the previous app version must exist from day one** regardless.
* **Secret rotation**: tied to the Credential Store cadence above; rotation is a pipeline step, not a manual one.
* **Database / schema migration policy** (closing the migration gap, AD-11): forward-only — every change is additive first (add nullable column → backfill → enforce constraint), never a destructive rollback of a deployed HANA schema. Each app version must remain compatible with the schema of the *previous* version during a rolling deploy, since old and new instances run side by side mid-rollout.
* **Deployment cadence & feature flags** (AD-26): daily deploys to DEV/TEST, weekly to UAT, on-demand (including hotfixes) to PROD initially. Risky features (Guest Checkout, Favorites notifications) sit behind a simple config-backed flag so they can be killed without a redeploy — rollback handles a bad deploy, flags handle a bad feature within a good deploy.
* **Coding standards** (AD-27): ESLint + Prettier enforced as a CI merge gate; CAP handlers stay thin — business logic lives in a `lib/<domain>` module per the Aggregate Boundaries section, not inline in the handler, so handlers don't accumulate logic over time.

# Scalability & Cost Alignment

v1 targeted a stack (HANA Cloud, full BTP) sized for enterprise scale while the stated NFR ("thousands of vehicles, thousands of customers") is comparatively modest. This mismatch is addressed explicitly rather than left implicit:

* **Right-size HANA Cloud** for the stated scale at launch; do not default to the largest available tier. Define scale checkpoints (e.g., >50,000 vehicles or >100 branches) as the trigger to re-evaluate sizing — not a fixed calendar date.
* **Caching** (concrete TTLs per AD-16, closing the "too shallow" gap):

| Cache | TTL | Invalidation |
|---|---|---|
| Catalog listing (search/browse results) | 5 minutes | Immediate bust on `VehicleSold`, `VehiclePublished`, `VehicleReserved`, `VehicleReleased`, `VehicleCheckoutStarted` (AD-42) |
| Branch reference data | 1 hour | Immediate bust on branch update (low-frequency writes) |

  A TTL-only cache would let a sold vehicle appear available for up to the full TTL window, which is not acceptable for a live marketplace — event-driven invalidation is mandatory, not optional, alongside the TTL. The bust list must cover every transition that changes catalog availability in either direction, not just the ones that make a vehicle unavailable — `VehicleReleased` (back to available) and `VehicleCheckoutStarted` (temporarily unavailable, mid-checkout) were the two missing in an earlier draft.
* **Search**: HANA Cloud native full-text search is sufficient at this scale (AD-7); a dedicated search service is a future decision, not a Phase 1 default.

# Data Architecture

## Data Classification (AD-30)

| Tier | Examples | Handling |
|---|---|---|
| Public | Catalog listings (make, model, price, images) | No restriction; cacheable (see Scalability) |
| Internal | Branch operational data, reporting aggregates | Internal access only, branch-scoped (AD-3) |
| Confidential | Customer PII (name, address, phone, email) | Anonymized in lower environments; subject to Retention Policy |
| Restricted | Payment tokens, PII-bearing audit rows | SAQ-A scope (AD-4); 7-year retention (AD-12) |

This ties the PCI, GDPR, and lower-environment-anonymization rules defined elsewhere into one scheme — a new field added later gets classified into one of these four tiers instead of an ad-hoc rule being invented for it.

## Legacy Data Migration & Cutover Strategy (AD-31)

The original business case (manual processes, spreadsheets, fragmented per-branch systems) implies existing data to migrate — no prior version of this document addressed it.

* **One-time batch ETL, not a live sync** — legacy systems are being retired, not integrated long-term.
* **Branch-by-branch rollout**, not big-bang: each branch is migrated, reconciled, and cut over independently.
* **Reconciliation report per branch** before its legacy system is decommissioned (vehicle counts match, no orphaned records, no duplicate customers).
* **Parallel-run window** (e.g. 2 weeks) per branch where Operators can cross-check AutoMarket against the legacy process before full cutover.
* **Branch-level rollback**: if a branch's migrated data fails reconciliation, that branch continues on its legacy process and re-attempts migration — this does not block other branches or the platform.
* **Marketing consent is not inherited**: imported contact lists require explicit re-confirmation of marketing opt-in before use — legacy consent records (if any) cannot be assumed GDPR-valid.
* **Transactional/historical data is a separate legal basis**: past orders, reservations, and delivery records migrate under contract-performance / legitimate-interest grounds, which do not require fresh consent — only the marketing-opt-in flag needs re-confirmation, not the underlying purchase history. Conflating the two would block a legitimate migration on a consent requirement that doesn't apply to it.

# Performance Budget & Capacity Model

## Latency Budget (AD-32)

The `<500ms` API NFR (v1) had no breakdown — closing that:

| Hop | Budget |
|---|---|
| Approuter routing/token check | ≤ 50ms |
| CAP handler (business logic + guards) | ≤ 150ms |
| HANA Cloud query | ≤ 200ms |
| Network/serialization overhead | ≤ 100ms |
| **Total** | **≤ 500ms** |

A regression is now attributable to a specific hop instead of "the API got slower." This budget applies to the **cache-miss path** — a cache hit (AD-16) is served from the Approuter/edge layer and never reaches the HANA query line at all; cache-hit and cache-miss are different paths, not the same request measured twice.

## Capacity / Load Model (AD-33)

Assumption, **flagged pending a real traffic projection from the business** (see Open Decisions): peak ~500 concurrent users, ~200 req/s at the stated launch scale. Load-test target is 2x peak (400 req/s) against UAT before each PROD promotion, tying into the Testability performance-testing requirement.

## Error Budget & Incident Response (AD-34)

The 99.5% availability SLO implies an error budget of ≈3.65 hours/month. That budget needs burn-rate alerting, not just a target:

| Window | Threshold | Action |
|---|---|---|
| Fast burn | 5% of monthly budget consumed within 1 hour | Page on-call |
| Slow burn | 10% of monthly budget consumed within 6 hours | Open a ticket |

Escalation path: Operator on-call → Engineering on-call → Incident Commander (Sev1 only). Documented as a runbook under `docs/runbooks/`, referenced from the Alert Notification Service.

# Product Analytics & Instrumentation

## Funnel Instrumentation (AD-35)

v1's business objectives ("increase online sales," "improve customer engagement") were never made measurable. Funnel events reuse the existing Domain Event backbone — no separate analytics pipeline. Per the Vehicle State Machine there are **two funnels, not one** (an earlier draft of this section showed only the reservation-led path — the same class of "single happy path" omission v1's original state machine diagram had):

**Direct purchase** (skips Favorite/Reservation):
```text
CatalogViewed → VehicleDetailViewed → CheckoutStarted → PaymentSucceeded → DeliveryConfirmed
```

**Reservation-led** (via Favorite, Reservation, or Offer):
```text
CatalogViewed → VehicleDetailViewed → FavoriteAdded → ReservationCreated
   → CheckoutStarted → PaymentSucceeded → DeliveryConfirmed
```

Both feed the Reporting/Analytics domain (AD-15) as conversion-rate dashboards, reported **separately** — collapsing them into one funnel would hide whether reservation-led customers convert at a different rate than direct purchasers, which is exactly the signal v1's "improve reservation management" objective needs.

# Open Decisions for Stakeholders

These cannot be resolved by engineering alone — they require product/business input before the corresponding section above can move from "decided" to "implemented":

1. **PSP vendor** (Stripe / Adyen / SAP Multi-Bank Connectivity) — cost and regional coverage tradeoff.
2. **True multi-tenant SaaS ambition** — does AutoMarket ever sell to multiple independent dealership companies? Determines if/when AD-2 is revisited and CAP MTX work is scheduled.
3. **Mobile push provider** (Firebase Cloud Messaging vs. SAP Mobile Services) — relevant once the Phase 3 mobile app is scoped; Notification Domain should add a push channel alongside email at that point.
4. **Target regions / jurisdictions** for initial launch — needed to finalize data residency and which GDPR-equivalent regimes apply.
5. **Legal confirmation of the 7-year Audit Log retention period** (AD-12) — the actual mandatory period is jurisdiction-dependent and must come from legal counsel, not be assumed by engineering.
6. **SLO → SLA promotion**: the Reliability Targets table states internal engineering targets; whether any of them (e.g. checkout success rate) become a *contractual* SLA with dealership customers is a commercial decision, not a technical one.
7. **Analytics platform horizon**: stay on HANA Cloud calculation views (AD-15) or plan a Phase 4+ move to SAP Datasphere — depends on how far reporting needs grow, which is a product roadmap question.
8. **Definition of "similar" for `SimilarVehicleListed`** (AD-18): attribute equality is the Phase 1 default, but whether to add price-range tolerance or trim-level matching is a product decision best made once real Favorites usage data exists.
9. **Exact TCO / pricing** (Business Architecture cost model): the cost-driver table identifies what scales cost; actual figures require an SAP account team quote, which is a procurement step, not an engineering one.
10. **Real traffic projection**: the AD-33 capacity model (~500 concurrent users, ~200 req/s) is an engineering placeholder, not a business-supplied number — replace it as soon as real projections exist, and re-run the load test target accordingly.
11. **Formal Architecture Review Board adoption**: AD-36's pre-go-live gate is intentionally lightweight (four role-based sign-offs); whether the organization needs a standing ARB with its own charter is a scale-dependent governance decision, not one this document can make for the business.
12. **Reservation/Offer validity window** (AD-41): defaults to 48 hours as an engineering placeholder; confirm with product whether that's right, or whether it should vary by vehicle category or price tier.
