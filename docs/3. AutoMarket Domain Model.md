## 1. Purpose

This document defines the business domain model of the AutoMarket platform: business entities, aggregate boundaries, relationships, business rules, and domain events. It is implementation-independent — no database or technical design decisions are included (those live in the CDS Data Model Document).

## 3. Ubiquitous Language

| Term | Definition |
|---|---|
| Vehicle | A sellable inventory unit tracked through the Vehicle State Machine, owned by exactly one Branch |
| Active Reservation | A Reservation in a non-terminal status — the target of the one-active-reservation-per-vehicle uniqueness constraint |
| Reservation | A time-bounded hold (48h default, AD-41) for vehicle inspection, created either directly by a customer/guest or as a result of an approved Offer |
| Test Drive | Scheduled vehicle test session — never a reservation |
| Offer | A customer-proposed price below list, subject to Operator/Manager approval; always requires authentication |
| Order | Confirmed purchase transaction |
| Payment | Financial transaction |
| Delivery | Vehicle handover process |
| Favorite | A customer's saved vehicle; doubles as the subscription list for re-engagement notifications |
| Branch | The organizational/physical unit that owns Vehicles and scopes ABAC access |
| Domain Event | An immutable fact dispatched at-least-once via the transactional outbox; never a command |
| Aggregate Root | An entity that owns its own consistency boundary and is the only entry point for changes within that boundary |

## 4. Aggregate Overview

| Aggregate Root |
|---|
| User |
| Branch |
| Vehicle |
| Pricing (Price History) |
| Reservation |
| Offer |
| TestDrive |
| Order |
| Payment |

Aggregate boundaries are intentionally separated to reduce coupling and improve scalability. They reference each other **by ID only** — never as nested/embedded collections. `Vehicle` in particular is kept deliberately small (it owns only its own state machine and fields) so that the row-lock taken during a state transition stays cheap; a Reservation, Offer, or TestDrive lifecycle change must never require locking Vehicle's full object graph.

**Cross-aggregate workflow coordination is choreography, not orchestration.** Two workflows require it today:

* **Offer approval**: `OfferApproved` → Reservation attempts `ReservationCreated` → on failure, `ReservationCreationFailed` → Offer reverts to `PENDING`, Operator notified.
* **Checkout**: `CheckoutStarted` → Payment processes through its own state machine → on success, Vehicle → `SOLD`; on failure, Vehicle reverts.

A central saga engine is only justified once a workflow needs more than 2-3 compensating steps.

## 5. Identity Domain

### User (Aggregate Root)

Represents a platform user.

**Attributes**: UserId, Email, FirstName, LastName, PhoneNumber, Status, CreatedAt, UpdatedAt.

**Business Rules**:

* Email must be unique.
* Users can have one or more roles.
* Disabled or locked users cannot authenticate.
* Account lockout: 5 failed login attempts within 15 minutes → 30-minute time-based lockout.
* MFA is mandatory for Operator/Manager/Admin; risk-based/optional for Customer at launch (mandatory MFA for Customer would conflict with the Guest Checkout friction-reduction goal).

### Role

**Values**: Admin, Manager, Operator, Customer.

---

## 6. Branch Domain

### Branch (Aggregate Root)

Represents a dealership location.

**Attributes**: BranchId, Code, Name, Address, City, Country, Region, Status.

`Region` exists for data residency — the BTP subaccount and HANA Cloud region must match the regulatory jurisdiction of the branches it serves.

**Business Rules**:

* Branch code must be unique.
* Inactive branches cannot sell vehicles.
* Every `Vehicle`, `Operator`, and `Manager` belongs to exactly one Branch. `Customer` does not belong to a branch.

---

## 7. Vehicle Domain

### Vehicle (Aggregate Root)

Represents a vehicle listed for sale.

**Attributes**: VehicleId, VIN, PlateNumber, Brand, Model, Year, Mileage, FuelType, Transmission, Color, Price, Currency, Status, BranchId.

### Vehicle Status

`Draft` → `ForSale` → `Reserved` → `PendingPayment` → `Sold` → `Delivered`, with `Archived` reachable from `Draft` or `ForSale`.

See the Implementation Architecture Document's State Machine Pattern section for the full, authoritative transition table with guards — it must never be simplified into a single forward chain.

### VehicleImage

Represents images associated with a vehicle. **Attributes**: ImageId, VehicleId, URL, SortOrder.

### Vehicle Business Rules

| Rule | Statement |
|---|---|
| V-1 | Sold vehicles cannot be reserved. |
| V-2 | Sold vehicles cannot receive offers. |
| V-3 | Sold vehicles cannot receive test drive requests. |
| V-4 | Pending-payment vehicles are unavailable for purchase by another customer. |
| V-5 | A vehicle's catalog visibility must be invalidated on **every** status-changing event in either direction — `VehicleSold`, `VehiclePublished`, `VehicleReserved`, `VehicleReleased` (a reservation/offer being cancelled or expired), and `VehicleCheckoutStarted` (entering `PendingPayment`). An earlier draft of this rule only covered the "becomes unavailable" direction and missed `VehicleReleased`/`VehicleCheckoutStarted`. |

## 8. Pricing Domain *(new)*

Kept intentionally thin in Phase 1.

### PriceHistory (part of the Pricing aggregate)

Represents the list-price history of a Vehicle.

**Attributes**: PriceHistoryId, VehicleId, Price, Currency, EffectiveFrom, ChangedBy.

**Responsibilities**:

* Holds the current list price and an append-only history of changes — Vehicle's own `Price`/`Currency` fields remain the live, queryable current price; Pricing is the audit trail behind it.
* Offer-vs-list-price comparison used by the Offer domain during review.
* Emits `VehiclePriceDropped` when the price decreases, consumed by Favorites/Notification (Section 15).

This is deliberately minimal so Phase 4 ("Vehicle Pricing Intelligence") has a domain to extend rather than retrofitting one later.

---

## 9. Favorites Domain

### Favorite

Represents a customer's saved vehicle.

**Attributes**: FavoriteId, CustomerId, VehicleId, CreatedAt.

**Business Rules**:

* **F-1**: A customer may save the same vehicle only once (enforced by a unique constraint on Customer + Vehicle).
* **F-2**: A Favorite is the subscription list for three re-engagement notification triggers: `VehiclePriceDropped`, `VehicleSold`, and `SimilarVehicleListed` (emitted when a newly published vehicle matches an existing Favorite's make + model + category + branch — attribute equality only; fuzzy/preference-based matching is Phase 4 scope, not this domain's).

## 10. Reservation Domain

### Reservation (Aggregate Root)

Represents a vehicle inspection reservation.

**Attributes**: ReservationId, CustomerId *(nullable — see Rule R-5)*, GuestToken *(nullable)*, VehicleId, BranchId *(denormalized from Vehicle at creation, see Rule R-6)*, ReservationDate, CreatedAt, Notes, Status.

### Reservation Status

`Requested` (or `Active`) → `Approved` → `Completed` / `Cancelled` / `Expired`.

### Business Rules

| Rule | Statement |
|---|---|
| R-1 | Only vehicles in `FOR_SALE` status can be reserved. |
| R-2 | Approved reservations change vehicle status to `RESERVED`. |
| R-3 | Cancelled or expired reservations return the vehicle to `FOR_SALE` (and emit `VehicleReleased`). |
| R-4 | Only one active reservation is allowed per vehicle — enforced by a partial unique constraint, not just an application check. |
| R-5 | A guest may create a Reservation without authentication (Guest Checkout). It carries a signed `GuestToken` instead of a `CustomerId`. At checkout, the guest authenticates or registers, which emits `ReservationClaimed` and sets `CustomerId`; from that point the reservation behaves exactly like a customer-created one. **Offer is not part of this rule** — Offer submission already requires authentication, so an Offer-derived Reservation always has a `CustomerId` from creation. |
| R-6 | `BranchId` is denormalized from the Vehicle at creation time so branch-scoped ABAC checks (Operators/Managers restricted to their assigned branch) evaluate against a direct column instead of a join through Vehicle. |
| R-7 | A Reservation's validity window is **48 hours**, anchored to its original `ReservationCreated`/`OfferApproved` timestamp — a failed checkout attempt does not reset the clock. This is an engineering default pending product confirmation; it may need to vary by vehicle category or price tier. |

## 11. Test Drive Domain

### TestDrive (Aggregate Root)

Represents a vehicle test drive request.

**Attributes**: TestDriveId, CustomerId, VehicleId, BranchId *(denormalized, same rationale as R-6)*, ScheduledDate, ScheduledTime, Status.

### Test Drive Status

`Requested` → `Approved` → `Completed` / `Cancelled`.

### Business Rules

| Rule | Statement |
|---|---|
| TD-1 | A test drive does not reserve a vehicle. |
| TD-2 | Vehicles remain available for sale during test drive scheduling. |
| TD-3 | Two test drives cannot occupy the same time slot for the same vehicle. |
| TD-4 | Customers must be notified if the vehicle is sold before the scheduled test drive. |
| TD-5 | Test Drive requests are allowed pre-authentication (Guest Checkout), same as Reservation — but a Test Drive never carries an ownership guard later in its lifecycle (a sold vehicle simply cancels it and notifies whatever contact info was captured at booking), so it needs no `GuestToken`/claim mechanism. |

---

## 12. Offer Domain

### Offer (Aggregate Root)

Represents a customer purchase proposal. **Always requires authentication** — it is not part of the Guest Checkout allow-list.

**Attributes**: OfferId, CustomerId, VehicleId, BranchId *(denormalized)*, OfferedPrice, DesiredPickupDate, Status.

### Offer Status

`Submitted` → `UnderReview` → `Approved` / `Rejected` / `Expired`.

### Business Rules

| Rule | Statement |
|---|---|
| O-1 | Offers can only be submitted for vehicles in `FOR_SALE` status. |
| O-2 | Approved offers generate a Reservation, which inherits the Offer's `CustomerId` directly (never a `GuestToken`, since the Offer already required authentication). |
| O-3 | Rejected offers may be resubmitted. |
| O-4 | Expired offers become read-only. |
| O-5 | An offer's approval validity window follows the same 48-hour default as Reservation (R-7) — it is the same underlying clock once the approval creates a Reservation. |

---

## 13. Sales Domain

### Order (Aggregate Root)

Represents a completed purchase process.

**Attributes**: OrderId, CustomerId, VehicleId, OrderDate, DeliveryType, Status.

**Delivery Type**: CustomerPickup, HomeDelivery.

**Order Status**: Created → PendingPayment → Paid → Completed / Cancelled.

### Business Rules

| Rule | Statement |
|---|---|
| S-1 | Only one active order can exist for a vehicle — must be backed by a DB-level uniqueness constraint in the CDS Data Model, not only stated here, since the corresponding Vehicle-level guard (V-4) is a separate enforcement point and either one being missing leaves a race window. |
| S-2 | Successful payment changes order status to `Paid`. |
| S-3 | Paid orders cannot be modified. |

---

## 14. Payment Domain

### Payment (Aggregate Root)

Represents a financial transaction.

**Attributes**: PaymentId, OrderId, Provider, Amount, Currency, TransactionReference, IdempotencyKey, Status.

### Payment Status

`Initiated` → `Authorized` → `Captured`, with `Failed` and `Refunded` as **separate terminal branches** — `Failed` only reachable from `Initiated`/`Authorized`, `Refunded` only reachable from `Captured`. A payment that succeeded cannot later become `Failed`; a payment that never captured cannot be `Refunded`.

### Business Rules

| Rule | Statement |
|---|---|
| P-1 | Payments are linked to exactly one order. |
| P-2 | Failed payments return the vehicle to `FOR_SALE` if no valid (non-expired) reservation exists, or to `RESERVED` if one does. |
| P-3 | Payment operations must be idempotent — every checkout attempt carries a client-generated idempotency key. |
| P-4 | AutoMarket never receives, processes, or stores raw card/PAN data. The Payment Service Provider handles tokenization; AutoMarket stores only a token, last4, and brand, keeping PCI-DSS scope at SAQ-A. |

---

## 15. Delivery Domain

### Delivery (Aggregate Root)

Represents the vehicle handover process.

**Attributes**: DeliveryId, OrderId, DeliveryType, PlannedDate, DeliveredDate, Status.

**Delivery Status**: Planned → InProgress → Delivered / Failed.

### Business Rules

| Rule | Statement |
|---|---|
| D-1 | Delivery can only start after successful payment. |
| D-2 | Completed deliveries change vehicle status to `DELIVERED`. |

---

## 16. Notification Domain

Represents customer communications.

**Channels**: Email, SMS, Push.

**Notification Status**: Pending → Sent / Failed / Retry.

**Favorites-driven triggers** (subscribers to the events in Section 8/9): `VehiclePriceDropped`, `VehicleSold`, `SimilarVehicleListed` — each resolves the customers who favorited the relevant (or matching) vehicle and queues one notification per favorite, through the same queue used for offer/reservation notifications.

Retention: Notification Logs are kept 1 year (operational troubleshooting only).

---

## 17. Audit Domain

### AuditLog

Captures business-relevant changes.

**Attributes**: AuditId, EntityType, EntityId, Action, OldValue, NewValue, UserId, Timestamp.

Retention: 7 years (legal/financial trail — the exact mandatory period is jurisdiction-dependent and pending legal confirmation; 7 years is an engineering default, not a legal sign-off). GDPR's "right to be forgotten" is satisfied by pseudonymizing the customer reference on request, not by deleting the audit row — the legal trail must survive, the personal identifier does not.

---

## 18. Reporting Domain

The reporting domain is a **read-only Analytical Model**, structurally separate from the **Operational Model** (the per-domain CAP services) — the two must never share a query path.

It is populated **asynchronously** from the same transactional outbox that feeds Notification and Audit, not by querying Vehicles/Orders/Payments/Reservations/Offers directly — that would couple reporting load to the transactional hot path.

**Retention is independent of the raw event stream**: Domain Events expire after 2 years, but the Reporting schema's own aggregates are not deleted on that schedule — purging raw events must never silently break a Manager's year-over-year comparison.

Typical reports: Vehicle Sales Report, Reservation Conversion Report, Offer Conversion Report, Branch Performance Report.

**Funnel instrumentation** feeds this domain as two separate funnels, not one (collapsing them would hide whether reservation-led customers convert at a different rate than direct purchasers):

* **Direct purchase**: `CatalogViewed → VehicleDetailViewed → CheckoutStarted → PaymentSucceeded → DeliveryConfirmed`
* **Reservation-led**: `CatalogViewed → VehicleDetailViewed → FavoriteAdded → ReservationCreated → CheckoutStarted → PaymentSucceeded → DeliveryConfirmed`

## 20. Domain Events

### Vehicle Events

`VehicleCreated`, `VehiclePublished`, `VehicleReserved`, `VehicleCheckoutStarted`, `VehicleReleased`, `VehicleSold`, `VehicleDelivered`, `VehicleArchived`.

`VehicleCheckoutStarted` and `VehicleReleased` close a gap found during reconciliation: every other Vehicle status had a corresponding event except entry into `PENDING_PAYMENT`, and `VehicleReleased` existed but had never been wired to anything (see Section 7, Rule V-5).

### Pricing Events

`VehiclePriceDropped`.

### Reservation Events

`ReservationCreated`, `ReservationClaimed` *(new — guest → identified customer, R-5)*, `ReservationApproved`, `ReservationCompleted`, `ReservationCancelled`, `ReservationExpired`.

### Offer Events

`OfferSubmitted`, `OfferApproved`, `OfferRejected`, `OfferExpired`, `OfferRequiresReReview` *(new — compensating event when an approved Offer's Reservation creation fails, see Section 4)*.

### Payment Events

`PaymentInitiated`, `PaymentSucceeded`, `PaymentFailed`, `PaymentRefunded`.

### Test Drive Events

`TestDriveRequested`, `TestDriveApproved`, `TestDriveCompleted`, `TestDriveCancelled`.

### Funnel / Analytics Events

`CatalogViewed`, `VehicleDetailViewed`, `FavoriteAdded` — reuse the same outbox; not a separate analytics pipeline.

## 21. Domain Invariants

The following must never be violated:

1. A sold vehicle cannot be purchased again.
2. A sold vehicle cannot be reserved.
3. Only one active reservation may exist per vehicle (DB-level constraint, not just application logic).
4. Only one active order may exist per vehicle (must be backed by a DB-level constraint — see Rule S-1).
5. Approved offers create reservations, which inherit the Offer's identity directly (never a guest token).
6. Payment transactions must be idempotent.
7. Test drives never reserve vehicles.
8. A guest-created Reservation cannot proceed past `CheckoutStarted` without first being claimed by an identified customer (Rule R-5).
9. A Reservation past its 48-hour validity window cannot be used to start checkout — it must already have transitioned to `EXPIRED`.
