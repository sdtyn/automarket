## 1. Purpose

This document defines how the AutoMarket solution shall be implemented at source code level.

The objective is to ensure:

* Consistent project structure
* Separation of concerns
* High maintainability
* Testability
* Future microservice readiness
* SAP CAP alignment

This document is the primary reference for developers during implementation.

---

## 2. Architecture Style

The implementation follows:

```text
Modular Monolith
+
DDD Lite
+
Clean Architecture
+
CAP Friendly
```

Per **AD-1**, this is one CAP project, one deployable. Domains are separated by namespace/folder, not by service boundary — splitting into microservices is deferred until a specific domain (Search, Notification, Payment) needs independent scaling or ownership.

Design Goals:

* Modular business domains
* Thin CAP service handlers
* Explicit business use cases
* Domain-driven code organization
* Independent testing
* Event-driven extensibility

## 3. Production Topology and Architectural Layers

Per **AD-9/AD-10**, no traffic reaches CAP directly. The full path:

```text
Browser
  │  HTTPS
  ▼
Application Router (AD-9)
  │  serves compiled UI5/Fiori apps from
  │  HTML5 Application Repository (AD-10)
  │  terminates OAuth2/XSUAA token exchange
  │  owns the session cookie + idle timeout
  ▼
CAP API Layer
  ▼
Application Layer
  ▼
Domain Layer
  ▼
Infrastructure Layer
  ▼
SAP HANA Cloud / XSUAA / Email / Payment Provider
```

**Network & region rules** (AD-22, AD-23):

* CAP is reachable only through the Approuter's route; HANA Cloud is reachable only via CAP's service binding. Nothing else is publicly routable.
* Single region for Phase 1 — multi-region is deferred until a target-region decision requires it (see Open Decisions in the Solution Architecture Document).

**Session timeout** (owned by the Approuter): Admin 15 min idle, Manager/Operator 30 min, Customer follows the IAS default.

## 9. API Layer

Location: `modules/*/api`

Purpose: expose OData services.

Example:

```text

  vehicle-service.cds
  vehicle-handler.js
```

Responsibilities: input validation, authorization checks, request orchestration.

Must NOT contain: business rules, database queries, state transitions.

Rate limiting (**AD-24, AD-39**) is enforced one layer up, at the Approuter — see Section 27.

## 12. Domain Layer

Location: `modules/*/domain`

Purpose: business rules and domain behavior.

Example:

```text
vehicle/domain
  Vehicle.js
  VehicleStateMachine.js
  VehicleRules.js
  VehicleEvents.js
  VehicleErrors.js
```

Responsibilities: state transitions, business invariants, domain validations, domain events.

Example methods: `canReserve()`, `canPurchase()`, `canSubmitOffer()`, `canScheduleTestDrive()`.

Per **AD-17/AD-19**, `Vehicle`, `Reservation`, `Offer`, `TestDrive`, `Order`, and `Payment` are each their own Aggregate Root, referencing each other by ID only — never nested. Cross-aggregate workflows (Offer approval, Checkout) are coordinated by **choreography**, not a saga engine: each aggregate reacts to events from the others (see Section 18).

## 13. State Machine Pattern

State transitions must be isolated in `VehicleStateMachine`. The transition table below is **authoritative** — it must not be simplified into a single forward chain, which was the exact error found and corrected in the Solution Architecture Document's first review pass.

### Vehicle State Machine

| From | To | Trigger | Guard |
|---|---|---|---|
| DRAFT | FOR_SALE | `VehiclePublished` | required fields complete (price, images, branch) |
| FOR_SALE | RESERVED | `ReservationCreated` | no active reservation exists for the vehicle |
| FOR_SALE | RESERVED | `OfferApproved` | same guard as above |
| FOR_SALE | PENDING_PAYMENT | `CheckoutStarted` | vehicle not currently RESERVED by another customer |
| RESERVED | PENDING_PAYMENT | `CheckoutStarted` | requester is the reservation/offer owner (or has just claimed it, AD-40); reservation within its 48h validity window (AD-41) |
| RESERVED | FOR_SALE | `ReservationCancelled` / `ReservationExpired` | — |
| PENDING_PAYMENT | SOLD | `PaymentSucceeded` | — |
| PENDING_PAYMENT | FOR_SALE | `PaymentFailed` (from direct purchase) | — |
| PENDING_PAYMENT | RESERVED | `PaymentFailed` (from a reservation/offer) | reservation still within validity window |
| SOLD | DELIVERED | `DeliveryConfirmed` | — |
| FOR_SALE / DRAFT | ARCHIVED | `VehicleArchived` | operator/manager action; vehicle removed from active sales |

`RESERVED` and `PENDING_PAYMENT` are **not** mandatory waypoints — a direct purchase legitimately skips `RESERVED`.

Invalid transition example (must throw a domain exception):

```text
SOLD
 ↓
FOR_SALE         ← invalid, throws DomainException
```

### Payment State Machine

| From | To | Trigger |
|---|---|---|
| INITIATED | AUTHORIZED | PSP authorization succeeded |
| INITIATED | FAILED | PSP authorization declined or timed out |
| AUTHORIZED | CAPTURED | Capture succeeded on order confirmation |
| AUTHORIZED | FAILED | Capture failed or authorization expired |
| CAPTURED | REFUNDED | Refund issued (full or partial) |

`FAILED` and `REFUNDED` are separate terminal branches — a captured payment cannot become `FAILED`, and a failed payment cannot be `REFUNDED`.

## 15. Infrastructure Layer

Location: `modules/*/infrastructure`

Purpose: technical implementations.

Examples: `VehicleRepository`, `OfferRepository`, `ReservationRepository`.

Responsibilities: persistence, external integrations, messaging, caching.

**Concurrency mechanics live here** (AD-6):

* `VehicleRepository` issues `SELECT ... FOR UPDATE` on the vehicle row before the domain layer evaluates a transition guard, closing the read-check-write race window. This is a short-lived row lock, not application-level pessimistic locking.
* Optimistic locking via CAP's `@odata.etag` on `modifiedAt` rejects conflicting concurrent updates at the HTTP layer (`412 Precondition Failed`).
* A partial unique constraint (`UNIQUE (vehicle_ID) WHERE status = 'ACTIVE'`) on Reservations is defense in depth against guard-logic bugs.
* Payment idempotency: every checkout attempt carries a client-generated idempotency key, stored unique per order.

**Background pollers** also live in the infrastructure layer:

* **Outbox dispatcher** (AD-5): reads unprocessed `EventOutbox` rows and dispatches them to subscribers — at-least-once delivery, consumers must be idempotent.
* **Reservation expiry job** (AD-41): scans `ACTIVE` reservations past `createdAt + 48h` and fires `ReservationExpired`. Anchored to the original timestamp — a failed checkout attempt does not reset the clock.

## 18. Event Architecture

Domain events belong to their modules and are dispatched through the **transactional outbox** (AD-5) — written in the same DB transaction as the state change, then dispatched at-least-once by the poller in Section 15.

```text
vehicle/domain/events
  VehicleCreated
  VehiclePublished
  VehicleReserved
  VehicleCheckoutStarted   ← added; every other Vehicle status had an event except entry into PENDING_PAYMENT
  VehicleReleased
  VehicleSold
  VehicleDelivered
  VehiclePriceDropped      ← Pricing domain, feeds Favorites notifications (AD-18)

offer/domain/events
  OfferSubmitted
  OfferApproved
  OfferRejected
  OfferExpired

reservation/domain/events
  ReservationCreated
  ReservationClaimed       ← guest → identified customer (AD-40)
  ReservationCancelled
  ReservationExpired
  ReservationCompleted
```

Events must NOT be stored in shared folders.

**Choreography, not orchestration** (AD-19): two workflows already require cross-aggregate coordination —

* **Offer approval**: `OfferApproved` → Reservation attempts `ReservationCreated` → on failure, `ReservationCreationFailed` → Offer reverts to `PENDING`, Operator notified.
* **Checkout**: `CheckoutStarted` → Payment processes `INITIATED → AUTHORIZED → CAPTURED` (or `FAILED`) → on `PaymentSucceeded`, Vehicle → `SOLD`; on `PaymentFailed`, Vehicle reverts per the state table above.

A central saga engine is introduced only if a workflow needs more than 2-3 compensating steps — neither does today.

## 20. Authentication Architecture

Production:

```text
SAP IAS → XSUAA → CAP
```

Local Development:

```text
Email → Password → JWT → CAP
```

**Guest flow** (AD-40, Guest Checkout): browsing, Reservation creation, and Test Drive requests are allowed pre-authentication. A guest-created Reservation carries a signed `guestToken` instead of a `customer_ID`. At `CheckoutStarted`, full identity is required: the guest authenticates or registers, which emits `ReservationClaimed` and sets `customer_ID`. **Offer submission is not part of this flow** — it already requires authentication, so it never carries a `guestToken`.

## 22. Logging Architecture

Location: `infrastructure/logging`

Log Levels: `ERROR`, `WARN`, `INFO`, `DEBUG`.

Rules: business exceptions must be logged; technical exceptions must be logged; sensitive information must never be logged.

Retention is enforced downstream, not at the logging layer (see Audit and Notification module documentation): Audit Logs 7 years, Domain Events 2 years, Notification Logs 1 year (AD-12). Reporting/Analytics projections derived from Domain Events are retained independently of that 2-year window (AD-37).

## 23. Testing Architecture

```text
test/
  unit/
  integration/
  contract/
  e2e/
  load/
  chaos/        ← new
```

| Layer | Tests | Target share |
|---|---|---|
| Unit | State Machines, Domain Services, Use Cases | ~70% |
| Integration | Repositories, CAP Services, Database interactions | ~20% |
| Contract | OData APIs, event payload schemas, service compatibility | — |
| E2E | Customer journeys: reservation flow, purchase flow, offer flow, guest-to-claim flow | ~10% |
| Load | k6/JMeter against the latency budget (Section 26) and SLO targets | — |
| Chaos | Kill the outbox dispatch poller mid-batch (non-prod) to verify at-least-once delivery and consumer idempotency hold under failure (AD-29) | — |

**Coverage** (AD-28): 80% minimum on domain/business-logic modules (state machines, guards, `VehicleAvailabilityService`), 60% overall — enforced as a CI merge gate, not just measured.

## 25. Coding Standards

Naming: `ReserveVehicleUseCase`, `VehicleRepository`, `VehicleStateMachine`.

Avoid: `VehicleManager`, `VehicleHelper`, `VehicleProcessor`. Class names must describe business intent.

**Enforcement** (AD-27): ESLint + Prettier as a CI merge gate. CAP handlers stay thin per Section 10 — this is not a style preference, it's how the Domain layer's aggregate boundaries (Section 12) stay intact as the codebase grows.

## 26. Performance Budget

The `<500ms` API NFR (Solution Architecture Document) is broken down by hop (AD-32), applying to the **cache-miss** path — a cache hit is served from the Approuter/edge layer and never reaches the HANA query line:

| Hop | Budget |
|---|---|
| Approuter routing/token check | ≤ 50ms |
| CAP handler (business logic + guards) | ≤ 150ms |
| HANA Cloud query | ≤ 200ms |
| Network/serialization overhead | ≤ 100ms |
| **Total** | **≤ 500ms** |

**Capacity assumption** (AD-33, flagged pending a real traffic projection from the business): peak ~500 concurrent users, ~200 req/s. Load-test target is 2x peak (400 req/s) against UAT before each PROD promotion.

## 27. Network, Deployment & Release

**BTP subaccount topology** (AD-21):

```text
Global Account
  ├─ Subaccount: DEV   (shared, pooled entitlement)
  ├─ Subaccount: TEST  (shared, pooled entitlement)
  ├─ Subaccount: UAT   (own entitlement)
  └─ Subaccount: PROD  (full entitlement, isolated quota)
```

**Schema migration policy** (AD-11): forward-only — additive first (add nullable column → backfill → enforce constraint), never a destructive rollback. Each app version stays compatible with the previous version's schema during a rolling deploy.

**Deployment cadence & feature flags** (AD-26): daily to DEV/TEST, weekly to UAT, on-demand to PROD. Risky features (Guest Checkout, Favorites notifications) sit behind a config-backed flag so they can be killed without a redeploy.

**Rate limiting** (AD-24, AD-39), enforced at the Approuter/API layer:

| Scope | Limit |
|---|---|
| Authenticated user — writes | 100 req/min |
| Authenticated user — reads | 300 req/min |
| Anonymous/guest — writes (Reservation, Test Drive) | 20 req/min per IP |
| Anonymous/guest — reads | 100 req/min per IP |

## 29. Architecture Compliance Rules

Prohibited:

* Business logic in CAP handlers
* Database access from UI
* Direct repository access from UI
* Cross-module database queries
* Shared business rules
* A guest-created resource being readable/cancellable without presenting its `guestToken` or the claiming customer's identity (AD-40)

All business logic must pass through `API → Application → Domain → Infrastructure`.
