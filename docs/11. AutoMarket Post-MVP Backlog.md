**Goal:** Fix the bugs discovered and deliberately deferred while writing EPIC15/EPIC16 tests and
while auditing `NotificationService`, all logged in `docs/error-log.md` — before building any new
feature (price-drop alerts, UI, production readiness) on top of them.

| # | Ticket | Description | Status |
|---|---|---|---|
| EPIC17-T1 | `retryPayment` / Order `CANCELLED` design fix | Resolve the EPIC08/EPIC09 contradiction: `failPayment`'s `PaymentFailed` choreography always sets `Orders.status = CANCELLED`, but `retryPayment` requires `PENDING_PAYMENT` — so retry can never run. Decide and implement a fix (either the `PaymentFailed` handler leaves a retryable state, or the `retryPayment` guard changes) | Open |
| EPIC17-T2 | `NotificationService`: `VehiclePriceDropped` wired to the wrong service | `notification-service.js` subscribes via `VehicleSrv.on('VehiclePriceDropped')`, but the event is declared and emitted only by `PricingService` — the listener never fires. | EPIC17-T3 | `NotificationService`: `resolveUserId` email/UUID mismatch | `resolveUserId` looks up `Users` by `email`, but `customer_ID` fields across the app (`Favorites`, `Orders`, ...) store `req.user.id`, which is the `Users.ID` UUID — so the lookup always returns `null` and no notification is ever created for `VehicleSold`, `VehiclePriceDropped`, or `SimilarVehicleListed`. Fix the lookup to match on `ID` | Open |
| EPIC17-T4 | Regression test | Integration test proving all three `NotificationService` subscribers (`VehicleSold`, `VehiclePriceDropped`, `SimilarVehicleListed`) now create a `Notification` row for a favoriting customer | Open |

**DoD:** All three known issues are fixed or a deliberate design decision is recorded; a
regression test proves notifications are actually created; CI stays green.

## EPIC20 — Full UI & Backend Integration

**Goal:** EPIC19 only wired browse/view screens (Vehicles, Users, Branches, AuditLogs) — every
actual business workflow (reserve, offer, test-drive, checkout, pay, approve, admin actions) is
still API-only, because almost every write operation in this system is an *unbound* OData action
(service-level, not bound to an entity type), and `@UI.DataFieldForAction` — Fiori Elements'
native toolbar-button mechanism — only targets actions bound to an entity type. This epic converts
the ~20 unbound actions that matter for a real end-to-end demo into bound actions (backend) and
wires each one onto the relevant List Report/Object Page as a native button (UI), ticket by
ticket, verified with the same rigor as EPIC19 (live backend + live `ui5 serve`, not just
annotations read for plausibility). Backend and UI land together in every ticket — no ticket is
"just the refactor" or "just the button."

## EPIC21 — Fiori Elements Multi-App Remediation

**Goal:** Fix a defect discovered via real-browser (Playwright) testing during EPIC20 sign-off,
documented in `docs/cap-notes.md` #12: `sap.fe.templates` does not support hosting multiple
unrelated List Report/Object Page pairs in one `sap.fe.core.AppComponent` the way EPIC19-T5/T6 and
EPIC20-T1 through T6 assumed ("manually merge the Nth entity into an existing app via extra
manifest.json routes"). Every entity beyond an app's original root (`Vehicles` in
`app/operator-portal`/`app/customer-portal`, `Users` in `app/admin-portal`) crashes to a full-page
"Sorry, we can't find this page" error the instant its route is opened — confirmed for
`ReservationsList`/`TestDrivesList`/`OffersList` (operator), `PaymentsList` (admin), `OrdersList`
(customer), and expected identically for `BranchesList`/`AuditLogsList` (admin, EPIC19-T5/T6) and
every remaining customer-portal entity by the same construction. An attempted
`sap.fe.core.rootView.Fcl` fix stops the crash but breaks the List Report's search/"Go" action for
every non-root entity instead — not a real fix. The only pattern proven to work end to end is one
entity's List Report as the sole root of its own app. This epic splits every affected entity into
its own dedicated Fiori Elements application, and separately fixes the native "Create" toolbar
button (EPIC20-T4's core deliverable) never appearing on `Vehicles` despite the `CREATE` grant,
because CAP is not emitting the `Capabilities.InsertRestrictions` annotation Fiori Elements needs
to decide to show it. The backend (CDS bound actions, JS handlers, `@UI` annotations) built in
EPIC19-T5/T6 and EPIC20-T1–T6 is correct and unaffected — this epic is UI-hosting-structure only.

| # | Ticket | Description | Status |
|---|---|---|---|
| EPIC21-T1 | Operator Portal — split into separate apps | Extract `Reservations`, `TestDrives`, `Offers` out of `app/operator-portal` into their own dedicated Fiori apps (own `manifest.json`, `Component.js`, `index.html`, `i18n`, `ui5.yaml`, `flpSandbox.html`), mirroring the proven working `VehiclesList`-as-sole-root pattern | Open |
| EPIC21-T2 | Admin Portal — split into separate apps | Same extraction for `Branches` (EPIC19-T5), `AuditLogs` (EPIC19-T6), and `Payments` (EPIC20-T5) out of `app/admin-portal` | Open |
| EPIC21-T3 | Customer Portal — split into separate apps | Same extraction for `Reservations`, `Offers`, `TestDrives`, `Orders`, `Payments` out of `app/customer-portal` (EPIC20-T1–T3) | Open |
| EPIC21-T4 | Native Create button fix | Add `Capabilities.InsertRestrictions` (or equivalent CAP config) so entities with a `CREATE` grant — starting with `OperatorPortalService.Vehicles` (EPIC20-T4) — actually show a native "Create" toolbar button in Fiori Elements, verified in a real browser | Won't Fix — requires CAP draft mode (see `docs/cap-notes.md` #13); `POST /operator/Vehicles` still works without the toolbar button |
| EPIC21-T5 | Per-role navigation | Once each entity is its own app, give every role a way to actually reach all of theirs — either a real Fiori Launchpad (`sap.ushell`) config or consolidated `flpSandbox.html` tiles per role — replacing the single-app-many-tiles assumption T1–T6 shipped with | Open |

**DoD:** Every List Report/Object Page delivered in EPIC19-T5/T6 and EPIC20-T1–T6 is reachable and
functional (loads real data on "Go", buttons render and work) when clicked through in a real
browser — verified with Playwright/`chromium-cli`, not just backend curl calls and `$metadata`
grep counts. The native "Create" button is visible and functional on `Vehicles`.

## EPIC24 — SAP BTP Deployment

**Goal:** Actually deploy the application to a real SAP BTP subaccount. EPIC23 made the app deployable in principle (Dockerfile, PostgreSQL adapter, XSUAA config, env-var-driven secrets) but stopped short of BTP's own deployment shape — no approuter, no deployment descriptor, no UI deployment strategy, and no CI/CD step that actually deploys anywhere. Identified while explaining to the user what a real BTP deployment would require (2026-07-09) — not yet scoped into tickets with estimates, captured here so the gap isn't lost.

| # | Ticket | Description | Status |
|---|---|---|---|
| EPIC24-T1 | Complete the Approuter | `approuter/` has a stub `xs-app.json` and a rate-limiting policy doc (AD-24) but no `package.json`/`@sap/approuter` dependency — not a deployable unit yet. Wire it up as the real single entry point: XSUAA-authenticated routing to the backend and to each UI app, CSRF protection (already declared in `xs-app.json`, unverified), and the already-documented rate-limiting policy actually implemented (currently deferred, per `approuter/rate-limiting.md`) |
| EPIC24-T2 | Deployment descriptor | `mta.yaml` (Cloud Foundry) or Kubernetes manifests (Kyma) — whichever target is chosen — describing every module (backend, approuter, each UI app) and every service instance (`xsuaa`, `postgres`) plus their bindings. Nothing like this exists in the repo today |
| EPIC24-T3 | UI app deployment strategy | Decide and implement: bundle `app/*/webapp` into the backend's own container (current Dockerfile's approach, dev-pragmatic but not BTP-native) vs. deploy each to the HTML5 Application Repository and serve via the Approuter (the standard BTP pattern). Whichever is chosen, update the Dockerfile/deployment descriptor accordingly |
| EPIC24-T4 | Real BTP service provisioning | Actually run the `cf create-service`/`cf bind-service` sequence already documented in `docs/dev-notes.md` §4 (XSUAA) and the PostgreSQL equivalent (T1) against a real BTP subaccount — closes the "not verified against a real BTP subaccount" gap flagged throughout EPIC23 (cap-notes.md #20, dev-notes.md §3–4) |
| EPIC24-T5 | CI/CD deploy step | Extend `.github/workflows/ci.yml` (currently builds + pushes the image to GHCR and stops, EPIC23-T5) with an actual `cf push`/`kubectl apply` step against the target chosen in T2, gated behind a real deployment environment/approval |

**DoD:** The application is reachable at a real BTP URL, behind the Approuter, with a genuine XSUAA login flow (not mocked auth) and a real bound PostgreSQL instance. A GitHub Actions run on merge to main deploys the new version automatically (or on manual approval, if a staging gate is wanted).
