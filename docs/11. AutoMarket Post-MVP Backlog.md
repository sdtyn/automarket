## EPIC17 — Known Issue Remediation

**Goal:** Fix the bugs discovered and deliberately deferred while writing EPIC15/EPIC16 tests and
while auditing `NotificationService`, all logged in `docs/error-log.md` — before building any new
feature (price-drop alerts, UI, production readiness) on top of them.

| # | Ticket | Description | Status |
|---|---|---|---|
| EPIC17-T1 | `retryPayment` / Order `CANCELLED` design fix | Resolve the EPIC08/EPIC09 contradiction: `failPayment`'s `PaymentFailed` choreography always sets `Orders.status = CANCELLED`, but `retryPayment` requires `PENDING_PAYMENT` — so retry can never run. Decide and implement a fix (either the `PaymentFailed` handler leaves a retryable state, or the `retryPayment` guard changes) | Open |
| EPIC17-T2 | `NotificationService`: `VehiclePriceDropped` wired to the wrong service | `notification-service.js` subscribes via `VehicleSrv.on('VehiclePriceDropped')`, but the event is declared and emitted only by `PricingService` — the listener never fires. Move the subscription to `PricingService` | Open |
| EPIC17-T3 | `NotificationService`: `resolveUserId` email/UUID mismatch | `resolveUserId` looks up `Users` by `email`, but `customer_ID` fields across the app (`Favorites`, `Orders`, ...) store `req.user.id`, which is the `Users.ID` UUID — so the lookup always returns `null` and no notification is ever created for `VehicleSold`, `VehiclePriceDropped`, or `SimilarVehicleListed`. Fix the lookup to match on `ID` | Open |
| EPIC17-T4 | Regression test | Integration test proving all three `NotificationService` subscribers (`VehicleSold`, `VehiclePriceDropped`, `SimilarVehicleListed`) now create a `Notification` row for a favoriting customer | Open |

**DoD:** All three known issues are fixed or a deliberate design decision is recorded; a
regression test proves notifications are actually created; CI stays green.

## EPIC18 — Price-Drop Alerts

**Goal:** Complete the `VehiclePriceDropped` notification flow. The event is already emitted by `PricingService`; the `NotificationService` subscriber stub is already wired (the wiring bug and the `resolveUserId` bug that block it are fixed by EPIC17 first). This EPIC fills in the remaining spec (channel, language), adds a user preference check, and covers the flow with integration tests.

| # | Ticket | Description | Status |
|---|---|---|---|
| EPIC18-T1 | NotificationService price-drop handler | Update the `VehiclePriceDropped` handler in `notification-service.js` to match spec: channel EMAIL, subject + content in German (currently PUSH / English, fixed in EPIC17-T2/T3) | Open |
| EPIC18-T2 | Notification preference field | Add `notifyOnPriceDrop: Boolean default true` to `Users` entity; filter handler to only notify users where preference is true; add `updateNotificationPreference` action to IdentityService | Open |
| EPIC18-T3 | Integration test | Add `price-drop-alert.http` to `tests/http/`: set a vehicle price lower → assert Notification row created for customer.bauer (who has that vehicle in Favorites) | Open |

**DoD:** Setting a vehicle price lower than its current price triggers a Notification row for each user who favorited it and has `notifyOnPriceDrop=true`. Verified via integration test.
